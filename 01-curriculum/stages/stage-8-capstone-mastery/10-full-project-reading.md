# 10 Full Project Reading

这一章的全项目阅读目标，不是再学一个新 API，而是把毕业项目当成一个完整产品，从目录、文件、主路径、风险点和下一版路线全部讲清楚。

## 一、全项目阅读顺序

建议按下面顺序扫完整个毕业项目：

1. 产品入口与用户旅程
2. 前端关键组件与交互主线
3. 后端 / API / runtime 主入口
4. Tools / RAG / memory / state / store 等核心能力层
5. Observability / evaluation / governance 层
6. 部署与环境配置层
7. 历史遗留、风险点与下一版方向

## 二、你至少要能找到哪些文件

毕业项目里，至少要能把这些职责块说清楚：

- `README.md` 或项目总览页
  项目定位、用户、主路径。
- `app/`、`frontend/`、`components/`
  前端界面和关键交互。
- `api/`、`routes/`、`server/`
  前后端边界、请求组织。
- `graph.py`、`agent.py`、`workflow.py`
  Agent runtime、图、主执行逻辑。
- `tools/`、`rag/`、`memory/`、`store/`
  外部能力、检索、记忆、持久化。
- `observability/`、`evaluation/`
  traces、评测、实验回流。
- `Dockerfile`、`.env.example`、部署配置
  运行环境和交付配置。

## 三、一条完整主路径应该怎么讲

毕业项目的主路径，通常至少要能被你讲成这样：

1. 用户从前端进入系统。
2. 前端如何组织请求并触发 agent 或 workflow。
3. 后端如何接住请求并装配上下文。
4. Agent runtime 如何决定调用哪些工具、知识或状态。
5. 中间结果如何被 stream、可视化、审批或恢复。
6. 最终结果如何返回并展示给用户。
7. traces、评测和治理信息如何在旁路被记录。

## 四、这一章必须做的全项目解释

做完毕业项目后，你至少要能完整解释：

- 哪个目录最能体现产品主价值。
- 哪个文件是系统真正的 runtime 入口。
- 哪个文件最能体现你的前端体验设计。
- 哪个文件最能体现你的工程化治理和观测。
- 哪 2 到 3 个文件最脆弱、最值得优先审查。
- 如果删掉某个关键文件，系统哪条主路径会断掉。

## 五、毕业项目要怎么做代码答辩

你必须能把这几类问题讲清楚：

- 为什么这个项目真的需要 Agent。
- 为什么当前架构用了 LangChain / LangGraph / Deep Agents 中的哪些层。
- 为什么某些功能没有在第一版实现。
- 哪些代码体现了你最关键的取舍。
- 哪些代码如果重来一遍，你会优先改。

## 六、最容易漏掉的全项目问题

- 只会展示功能，不会讲文件职责和调用链。
- 知道系统能跑，但说不清哪里最脆弱。
- 会解释主路径，但不会解释失败路径和恢复路径。
- 会讲当前版本，却讲不出下一版的演进方向。

## 七、本章的读码完成标准

如果你已经能：

- 沿目录讲清整个毕业项目的职责分层。
- 用一条主路径把最关键的文件串起来。
- 指出系统最脆弱的部分和最值钱的部分。
- 把“为什么这么实现”和“下一版怎么改”讲清楚。

那这一章的全项目理解才算完成。
