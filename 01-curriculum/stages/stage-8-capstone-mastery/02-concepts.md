# 02 Concepts

## 1. 毕业项目不是“大一点的作业”

在这套课程里，毕业项目不是简单把功能做多，而是要展示：

- 选题判断
- 架构能力
- 系统边界意识
- 产品思维
- 复盘能力

## 2. 综合覆盖的含义

一个毕业项目至少应该覆盖：

- Agent 主线
- 知识或工具能力
- 状态或工作流能力
- 工程化层
- 前端体验层
- 交付与评测层

覆盖不意味着每一层都做到最复杂，但必须都能解释清楚。

## 3. 毕业项目的价值标准

不是“功能多”，而是：

- 场景真实
- 用户明确
- 主路径清楚
- 架构有理由
- 风险被看见
- 下一版有方向

## 4. 答辩为什么重要

答辩的意义不是背答案，而是验证：

- 你能不能解释自己的设计。
- 你能不能面对质疑。
- 你能不能做出合理取舍。

## 5. 复盘为什么是毕业的一部分

如果一个人只能展示结果，不能解释：

- 哪些地方做得对
- 哪些地方做得不好
- 下一版要怎么改

那说明掌握还不稳定。

## 6. 方法迁移才是真正掌握

Stage 8 最终要验证的不是“这个项目做出来了”，而是：

- 你是否能把同样的方法迁移到别的行业、别的产品、别的约束条件里。

## 7. Stage 8 的结论

真正毕业，不是完成一个项目，而是形成一套能持续构建和迭代 Agent 产品的方法。

## 8. 代码教材：毕业项目应该怎么被讲清楚

这一节开始不再引入新框架，而是把前 1-7 阶段学到的内容，收束成“毕业项目代码该怎么读、怎么讲、怎么答辩”。

### 8.1 毕业项目的第一份代码产物不是实现，而是文件地图

真正能讲清项目的人，通常都会先有一张“项目文件地图”。

```text
app/
components/
api/
graph.py
agent.py
tools/
rag/
store/
observability/
evaluation/
Dockerfile
.env.example
```

这一层的价值不是列文件名，而是先把职责说清楚：

- 哪些文件面向用户体验
- 哪些文件面向运行时
- 哪些文件面向工程治理
- 哪些文件面向交付和运维

### 8.2 毕业项目的代码讲解必须从主路径开始

毕业答辩时，最关键的不是背模块清单，而是把一条真实用户路径讲顺。

```text
用户输入 -> 前端提交 -> API / server route -> agent runtime
-> tools / RAG / state -> result / interrupt / stream
-> 前端展示 -> traces / evaluation / audit
```

如果这条链讲不顺，说明项目还是停留在“做出来了”，还没到“理解清楚了”。

### 8.3 关键代码说明应该至少覆盖 3 种文件

毕业项目里最少要能挑出三类代码来讲：

1. 一个最能体现用户价值的文件  
   比如主界面、主入口、核心路由。

2. 一个最能体现 Agent 特性的文件  
   比如 `graph.py`、`agent.py`、核心 workflow。

3. 一个最能体现工程化成熟度的文件  
   比如 observability、evaluation、policy、deployment 配置。

这三类文件一起，才能证明你理解的不是局部，而是产品全貌。

### 8.4 毕业项目的审核重点不只是 bug

到了 Stage 8，代码审核至少要看四类东西：

- 主路径是否清楚
- 边界是否清楚
- 风险是否被看到
- 下一版演进点是否合理

也就是说，毕业项目的审核既是技术审核，也是产品审核和架构审核。

### 8.5 下一版方向本身也是代码理解的一部分

如果你真的理解了项目代码，你应该能说出：

- 哪部分是现在最脆弱的
- 哪部分是现在最值钱的
- 哪部分应该暂时不动
- 下一版如果只能改一件事，改哪里最划算

这说明你已经从“解释现有实现”，进入了“能推动系统迭代”。

### 8.6 这一节读完后你至少要会什么

你至少要能用自然语言解释：

- 毕业项目的目录和文件职责为什么这样组织
- 一条完整主路径是怎么穿过整个系统的
- 哪几个文件最能代表项目价值和技术深度
- 哪几个文件最值得优先审查
- 下一版为什么应该先改某个地方而不是别的地方
