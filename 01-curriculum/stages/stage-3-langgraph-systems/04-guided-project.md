# 04 Guided Project

`Stage 3` 的引导项目选择 `带人工审批的流程 Agent`，因为它最能体现 LangGraph 的真正价值: 不只是“能跑起来”，而是“知道什么时候暂停、什么时候恢复、为什么这样流转”。

这一章的重点不是把它做成企业级流程引擎，而是让学习者第一次真正把控制流、状态和审批逻辑设计成显式图。

## 项目定位

### 你要设计的是什么

你要设计一个任务执行工作流。系统需要:

- 接收任务
- 生成执行计划
- 在关键动作前等待审批
- 审批通过后继续执行
- 审批拒绝或执行失败时进入恢复路径
- 最终输出结果

### 你现在不追求什么

这一阶段故意不追求:

- 多 Agent 协作
- 超复杂业务规则
- 全量前端实现
- Deep Agents 级别任务规划

因为 `Stage 3` 真正要训练的是: 如何把一个有状态、有审批、有恢复的任务推进成正式图系统。

## 这个项目真正训练什么

这个项目表面上是审批流程，实际上训练的是 6 种能力:

- 主路径显式化
- state 设计
- interrupt 设计
- 恢复路径设计
- durable execution / persistence 意识
- 为什么这里必须用 LangGraph 的边界判断

## 最终交付物

完成这个引导项目后，你应该至少交出以下内容:

- 一张工作流图
- 一份 state 字段设计
- 一份 interrupt 设计说明
- 一份恢复策略说明
- 一份“为什么这里必须使用 LangGraph”的说明
- 一份运行时失败清单

## 项目步骤

## Step 1: 先画主路径

### 这一步要解决什么

先把系统的主骨架画清楚，而不是一开始就陷入所有分支。

### 你要回答的问题

- 从接收任务到输出结果，主路径怎么走
- 哪几个阶段是不可少的
- 哪个阶段真正构成图的骨架

### 推荐主路径方向

例如:

- 接收任务
- 生成计划
- 请求审批
- 执行动作
- 汇总结果

### 常见错误

- 还没画主路径就先画很多异常
- 把每个细节都画成节点，导致主骨架不清楚
- 图里全是动作，没有阶段感

### 这一小步的产出

- 一张主路径草图

## Step 2: 设计 state

### 这一步要解决什么

你要定义哪些运行事实必须被系统持续记住。

### 你要回答的问题

- 哪些信息会影响后续路径
- 哪些信息恢复时必须知道
- 哪些信息审批时必须展示

### 建议至少覆盖的字段

- 原始任务
- 当前阶段
- 当前计划
- 审批状态
- 执行结果
- 错误信息

### 常见错误

- 把大量无关上下文都塞进 state
- 只存任务输入，不存阶段信息
- 忘记存恢复路径真正需要的信息

### 这一小步的产出

- 一份 state 字段说明
- 一份“不放进 state 的内容”清单

## Step 3: 补关键分支

### 这一步要解决什么

你需要把真正影响路径的判断显式化，而不是让模型暗中决定。

### 你要回答的问题

- 审批通过和拒绝后分别去哪里
- 执行成功和失败后分别去哪里
- 输入不足时系统走哪条支路

### 常见错误

- 只有成功路径，没有拒绝或失败路径
- 把“分支”写成文字说明，没有真正进入图设计
- 让模型隐式决定是否继续，而图里看不见

### 这一小步的产出

- 一张补充分支后的图

## Step 4: 设置 interrupt

### 这一步要解决什么

你要决定系统在什么地方必须正式暂停。

### 你要回答的问题

- 哪些动作前必须等待审批
- 哪些信息缺失时不能继续
- 暂停后谁来决定继续还是回退

### 推荐 interrupt 位置

- 审批前
- 缺少关键输入时
- 方向需要用户确认时

### 常见错误

- 把 interrupt 设计成普通问答澄清
- 停在太晚的位置，已经越界后才停
- 停下来以后没有恢复路径

### 这一小步的产出

- 一份 interrupt 设计清单

## Step 5: 设计恢复路径

### 这一步要解决什么

你要让系统失败后还能继续，而不是直接作废。

### 你要回答的问题

- 审批拒绝后回到哪里
- 执行失败后回到哪里
- 输入补齐后从哪里恢复

### 常见错误

- 恢复路径只有“从头开始”
- 拒绝后没有回路，只能结束
- 失败后把责任全推给用户

### 这一小步的产出

- 至少 2 条恢复路径说明

## Step 6: 补运行时解释

### 这一步要解决什么

你要开始把 persistence、durable execution 和 streaming 的意义说出来，而不是只会画图。

### 你要回答的问题

- 为什么这个图需要 persistence
- 为什么审批后要能继续，而不是重来
- 前端应该看到哪些运行中状态

### 这一小步的产出

- 一段运行时能力说明
- 一份前端可见状态列表

## Step 7: 做边界复盘

### 这一步要解决什么

你现在要明确说明: 为什么这里不能只停留在 LangChain 高层原型。

### 你要回答的问题

- 为什么这个项目已经需要显式图编排
- 为什么 interrupt 不能只靠 prompt 模拟
- 为什么恢复能力在这里是主流程的一部分

### 这一小步的产出

- 一段“为什么必须使用 LangGraph”的说明

## Step 8: 用 Codex 做图系统评审

### 推荐你怎么配合 Codex

建议这样使用 Codex:

- 先帮我检查这张图里哪个分支最模糊，不要重画整张图。
- 只从 state 角度审查我的设计，不要帮我加新功能。
- 只从 interrupt 和恢复角度批评我的图。
- 帮我判断我是不是真的需要 LangGraph，而不是只是图画得更复杂。

### 一个很好的带学节奏

1. 你先回答一个步骤。
2. 让 Codex 指出 2 到 3 个关键问题。
3. 你修正后再继续。

## 阶段检查点

完成项目后，你至少应该能稳定说清楚:

- 这个系统的主路径是什么
- state 为什么这样设计
- interrupt 为什么停在这里
- 恢复为什么不是“从头再来”
- 为什么这个项目已经不适合继续停留在 LangChain 高层
