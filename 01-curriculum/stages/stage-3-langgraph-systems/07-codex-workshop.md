# 07 Codex Workshop

这个文件不是“给 Codex 的说明”，而是“给学习者的 Stage 3 协作手册”。它的目的不是让 Codex 替你把图画完，而是让你在和 Codex 的配合里，真正学会图系统设计。

## Stage 3 与 Codex 配合的核心原则

在这一阶段，Codex 最适合扮演这些角色:

- 图系统解释者
- state 审查员
- interrupt 审查员
- 恢复路径评审员
- LangChain / LangGraph 分界提醒员

在这一阶段，Codex 最不适合扮演的角色是:

- 直接替你完成整张工作流图的人
- 用更多节点掩盖结构问题的人
- 在你还没说明主路径前就把图越画越复杂的人

一句话原则:

> 在 Stage 3，Codex 应该帮你形成运行时系统判断，而不是替你跳过图系统设计。

## 使用前先告诉 Codex 什么

每次开始一个会话前，建议你至少告诉 Codex 这 5 件事:

1. 我现在处于 `Stage 3`。
2. 我的目标是学习图系统设计，不是直接拿完整图。
3. 请先问我问题，再帮我评审。
4. 不要一开始就替我铺满所有分支。
5. 如果你觉得我这个设计其实还停留在 Stage 2，请明确提醒我。

一个通用开场模板:

```text
我现在在学习 LangChain Agent Course 的 Stage 3。
这一阶段我想训练的是 LangGraph 图系统设计，不是直接拿完整答案。
请你先做导师和评审，不要直接给我完整工作流图。
每次只推进一步，先问我问题，再指出我的结构问题。
如果你觉得我这个项目其实还没到必须用 LangGraph 的程度，请明确告诉我原因。
```

## 模式一: 图系统概念讲解模式

适用场景:

- 你刚读完概念，但还不真正理解为什么这里需要图
- 你知道 state、interrupt、persistence 这些词，但讲不出自己的话
- 你还说不清 LangGraph 和 LangChain 的边界

推荐提问:

```text
请用课程学习者能听懂的方式解释为什么这里需要 LangGraph，而不是继续用高层原型。
不要只讲定义，要结合真实项目说明。
```

```text
请用一个审批流程案例解释 interrupt、persistence 和恢复之间的关系。
重点讲系统为什么需要它们，不要只讲 API。
```

```text
请解释 streaming 在图系统里的价值。
重点讲它为什么和前端可见性相关，而不只是“边生成边显示”。
```

## 模式二: 图式拆解模式

适用场景:

- 你已经有一个 Stage 2 原型，想把它升级成 LangGraph 设计
- 你不确定主路径、分支、state、interrupt 怎么定义
- 你想练图式设计四步法

推荐提问:

```text
我想把这个 Stage 2 原型升级成 LangGraph 工作流。
请先不要替我画完整图，先按主路径、state、分支、interrupt、恢复这几个维度一步一步带我拆解。
每一步先提问，不要直接替我填答案。
```

```text
请先判断这个项目现在是否真的需要 LangGraph。
如果还不需要，请明确告诉我最主要的原因。
```

## 模式三: state 审查模式

适用场景:

- 你总觉得 state 越写越多
- 你不确定哪些字段真的需要存在
- 你怀疑自己把很多无关信息都塞进了图里

推荐提问:

```text
请只从 state 角度审查我的设计。
告诉我哪些字段真正影响后续路径，哪些字段其实不该放进 state。
不要帮我加新字段。
```

```text
请帮我判断这里哪些字段是为了恢复必须保存，哪些只是“看起来有用”但现在不该进 state。
```

## 模式四: interrupt 与恢复评审模式

适用场景:

- 你不确定 interrupt 停得对不对
- 你担心所谓恢复其实只是重来
- 你想知道审批节点是不是放在正确位置

推荐提问:

```text
请只从 interrupt 与恢复角度审查我的图。
告诉我哪里应该停、哪里停得太晚、哪里恢复路径不够合理。
不要帮我增加新的业务能力。
```

```text
请判断我的恢复设计是不是其实只是在变相从头执行。
```

## 模式五: 图系统边界评审模式

适用场景:

- 你不确定自己是不是真的需要 LangGraph
- 你担心只是把项目画得更复杂
- 你想知道哪些节点和分支其实并不必要

推荐提问:

```text
请站在架构评审角度，判断这个项目为什么必须使用 LangGraph。
如果你觉得不够成立，请明确指出最薄弱的地方。
```

```text
请指出我这个图里哪些复杂度是真正必要的，哪些复杂度只是“画出来显得更像系统”。
```

## 模式六: 项目带学模式

适用场景:

- 你在做 [04-guided-project.md](./04-guided-project.md)
- 你在做 [05-challenges.md](./05-challenges.md)
- 你准备进入课后项目

推荐提问:

```text
请按 Stage 3 的知识点顺序带我完成审批工作流 Agent。
每次只推进一步，并先告诉我这一步对应哪个概念和方法。
不要一次给完整图。
```

```text
我现在做的是 Stage 3 课后项目。
请不要代做，只按主路径、state、interrupt、恢复和边界做导师和评审。
```

## 模式七: 模拟答辩模式

适用场景:

- 你准备做 Stage 3 放行检查
- 你想测试自己是否已经能清楚讲出图系统思路
- 你希望进入 Stage 4 前做一次运行时压力测试

推荐提问:

```text
你来当导师，逐题提问我 Stage 3 的答辩题。
不要先给答案，等我回答完再指出哪里不清楚。
```

```text
请重点追问我，为什么这个项目必须使用 LangGraph，以及 interrupt 和恢复在里面真正发挥了什么作用。
```

## 推荐学习动作

一个很稳的对练闭环是:

1. 先让 Codex 解释。
2. 再自己复述。
3. 再让 Codex 挑错。
4. 再修正你的表达。
5. 最后让 Codex 模拟图系统评审或答辩。

这个顺序会逼你从“看懂图系统”进入“能解释图为什么这样设计”。

## 什么时候不要让 Codex 直接回答

在下面这些场景里，最好不要让 Codex 直接给完整答案:

- 你正在做 challenge
- 你正在做课后项目
- 你准备做答辩
- 你还没有自己画过第一版图

更好的做法是:

- 先画你的版本
- 再让 Codex 做评审
- 最后自己再改一版

## 一个完整的 Stage 3 会话模板

你可以直接复制下面这段开始一轮高质量学习:

```text
我现在在学习 LangChain Agent Course 的 Stage 3。
我希望你扮演导师和评审，而不是直接代做。
请先根据我给你的项目，判断它是否真的已经需要 LangGraph。
如果需要，再按主路径、state、分支、interrupt、恢复这几个维度，一步一步带我拆解。
每次只推进一步。
先问我问题，再指出我回答里的结构问题。
不要直接替我画完整图，也不要一次给完整答案。
如果你觉得我这个项目其实还适合停留在 Stage 2，请明确告诉我原因。
```

## Stage 3 结束前，你应该达到的协作状态

如果这一章学得比较扎实，到了最后你和 Codex 的关系应该是:

- 你负责提出图系统草案
- Codex 负责挑战你的 state、interrupt 和恢复设计
- 你负责修正控制流和边界
- Codex 负责指出哪些复杂度是真正必要的
- 你最终能不依赖 Codex，也把一个 LangGraph 工作流讲清楚

这才说明 Codex 真正起到了“放大学习效率”的作用，而不是替你绕过了图系统思考过程。
