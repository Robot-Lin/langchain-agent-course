# 06 Review Checklist

这个文件的作用不是让你“打卡完成 Stage 3”，而是帮你判断: 你到底有没有真正掌握 LangGraph 的运行时思维，以及你是否已经准备好进入 `Stage 4`。

## 这一章怎么用

建议你分 3 次使用这个清单:

1. 学完概念和方法后，做第一次自测。
2. 做完引导项目后，检查图系统结构是否清楚。
3. 做完挑战题和课后项目后，做正式放行判断。

真正有效的用法不是机械打勾，而是要求自己: 每一项都能举出一个自己的例子，或者一个明确反例。

## 第一部分: 图系统概念理解自测

如果下面任意 3 项做不到，建议不要急着进入 `Stage 4`。

- 我能解释为什么复杂任务需要显式图编排。
- 我能解释 LangGraph 和 LangChain 的上下层关系。
- 我能说明 state 在图系统里的作用。
- 我能解释 interrupt 为什么是正式运行时暂停点。
- 我能说明 durable execution 与 persistence 的价值。
- 我能解释 streaming 为什么不仅是 UI 功能。
- 我能说出高层原型与图系统在控制流上的差别。
- 我能指出什么时候一个项目必须使用 LangGraph。

### 合格信号

- 你不是只会说“LangGraph 更适合复杂系统”
- 你能解释复杂在哪
- 你能说明为什么复杂到需要显式控制流

### 危险信号

如果你还有这些表现，建议回看概念部分:

- 把 graph 只理解成“画流程图”
- 把 interrupt 理解成普通对话澄清
- 把 persistence 理解成“顺便存一下历史”

## 第二部分: 图系统结构自测

请对你自己的一个方案做检查。如果下面这些问题答不清，说明你还没有真正掌握 LangGraph 设计。

- 我是否定义了清楚的主路径。
- 我是否显式定义了关键分支。
- 我是否设计了最小但够用的 state。
- 我是否说明了哪些点必须 interrupt。
- 我是否说明了每个 interrupt 后怎么继续。
- 我是否设计了至少 2 条恢复路径。
- 我是否说明了 persistence 在这里为什么必要。
- 我是否说明了前端应该看到哪些运行中状态。

### 合格信号

你的方案读起来应该有这种感觉:

- 主路径清楚
- 分支必要而不乱
- state 有边界
- interrupt 放得合理
- 恢复不是装饰

### 危险信号

如果你的方案里出现这些情况，说明图系统设计还不稳:

- 图上全是节点，但看不出主骨架
- state 越存越多，却说不清为什么
- interrupt 停在模糊位置
- 所谓恢复其实只是“从头再来”

## 第三部分: 风险与恢复自测

一个 `Stage 3` 合格学习者，不只会设计工作流，还应该能先看到运行时失败和恢复问题。

请检查自己是否能做到:

- 至少说出 5 个运行时失败点
- 能区分 state 问题、interrupt 问题、恢复问题、可见性问题
- 能指出至少 2 个最该人工介入的点
- 能说明为什么这些失败不能只靠 prompt 修补

### 合格信号

- 失败点具体，不抽象
- 改进思路针对运行时结构
- 能看到“用户不知道系统正在干什么”这类可见性问题

### 危险信号

- 所有问题都归因给模型
- 完全没有写恢复逻辑
- 没有说明 persistence 与恢复的关系

## 第四部分: 表达与复述自测

进入 `Stage 4` 前，你应该不看稿也能把一个 LangGraph 工作流讲清楚。

请检查自己是否能脱稿回答:

- 为什么这个项目必须使用 LangGraph
- state、interrupt、persistence 分别解决什么问题
- 为什么恢复不等于重新执行
- 为什么 streaming 和前端可见性相关
- 为什么这不是单纯“更复杂的 LangChain”

### 合格信号

- 回答里有运行时思维
- 有你自己的工作流例子
- 能解释“为什么暂停在这里”“为什么恢复到那里”

### 危险信号

- 只能讲术语，讲不出系统流转
- 一离开教材原句就不会说了
- 只能说“系统更复杂了”，讲不出复杂在哪

## 阶段答辩题

下面这组问题可以作为 `Stage 3` 的正式放行答辩题。

### 必答题

1. 哪些问题是 state 设计问题，而不是 prompt 问题。
2. interrupt 在这个项目里真正暂停了什么。
3. 为什么 durable execution 与 persistence 很重要。
4. 为什么恢复能力不能只理解成“失败后重试一次”。

### 进阶题

1. 为什么 streaming 在图系统里不仅是 UI 功能。
2. 为什么一个“能跑”的流程图仍然可能不适合上线。
3. 如果你把 Stage 2 的单工具研究助理升级到 Stage 3，你会先补什么。

### 合格标准

- 每题都能结合项目场景回答
- 每题都能提到运行时边界或失败风险
- 不只是会复述定义，而是真的会判断

## 项目产物检查

进入下一阶段前，建议你至少完成下面这些产物:

- 1 张 LangGraph 工作流图
- 1 份 state 设计说明
- 1 份 interrupt 设计说明
- 1 份恢复路径说明
- 1 组挑战题答案
- 1 次基于标准的自我答辩或 Codex 评审
- 1 份课后项目方案

如果这些产物都没有，说明你更像“读过 Stage 3”，而不是“掌握了 Stage 3”。

## 放行标准

只有当你已经能意识到下面这些事时，`Stage 4` 才会真正有效:

- 图不是装饰，而是控制流建模
- state 不是可有可无的补充
- interrupt 是正式运行时暂停点
- persistence 是恢复能力的重要基础
- 有些复杂任务必须先有 LangGraph，后面才谈更复杂的 agent 组织

更具体一点说，你至少要能做到:

- 不把所有复杂性继续塞回高层原型
- 不让图只有成功路径
- 不让恢复逻辑沦为“重来一次”
- 不忽略运行中状态可见性
- 能说清项目为什么必须使用 LangGraph

在进入 `Stage 4` 之前，先问自己一句:

如果你能明确回答后者，说明 `Stage 3` 的关键价值已经开始内化。
