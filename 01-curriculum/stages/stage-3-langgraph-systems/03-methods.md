# 03 Methods

本文件关注 Stage 3 的设计方法。

## 方法一: 图式设计四步法

1. 先画主路径
2. 再补关键分支
3. 再补失败与恢复
4. 最后补审批与人工介入

这样做的原因是:

- 主路径帮助你定住系统骨架
- 分支说明系统真的有决策结构
- 恢复和审批说明系统能在真实环境里工作

## 方法二: State 最小集设计法

每设计一个字段，都问:

- 这个信息会不会影响后续路径
- 这个信息是不是恢复执行时必须知道
- 这个信息是不是审批或回显时必须展示

如果三个问题都是否，先不要放进 state。

## 方法三: Interrupt 门设计

给工作流明确设置中断门。

常见位置:

- 调用高风险工具前
- 提交最终结果前
- 外部输入缺失时
- 任务方向需要用户确认时

Interrupt 的核心不是“停”，而是“停在最该停的地方”。

## 方法四: 恢复路径设计

不要只设计成功路径。

至少补三类恢复:

- 工具失败后的恢复
- 审批拒绝后的恢复
- 输入不足时的恢复

## 方法五: 图式复盘法

每次复盘都检查:

- state 是否过重或过轻
- 分支是否真的必要
- interrupt 是否放对位置
- 恢复是否只是“从头再来”
- 前端是否能看见执行状态

## 方法六: 从 LangChain 升级到 LangGraph 的判断法

如果项目开始出现下面情况，优先考虑升级:

- 你想明确控制流程，而不是让模型隐式决定
- 你需要正式暂停和恢复
- 你需要把任务拆成清晰节点
- 你需要图层级的可解释性

## Stage 3 方法目标

真正掌握本阶段，不是“会画图”，而是能把复杂任务的控制逻辑和状态逻辑设计成一个可靠系统。
