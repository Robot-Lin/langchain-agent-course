# 09 Case Comparisons

这个文件用于训练 `Stage 3` 的关键能力: 不是一看到复杂需求就盲目上图，而是先判断它是否真的已经进入 LangGraph 级别的问题。

建议你一边读，一边回答:

- 为什么这个场景已经需要图
- 为什么它的控制流不能再隐式交给模型
- 哪个点最需要 state、interrupt 或恢复

## 案例 1: 单工具研究助理升级版

### 场景

你原来有一个单工具研究助理，现在希望它能分阶段完成问题澄清、检索、总结和复审。

### 当前适合 LangGraph 吗

开始适合，尤其当阶段边界已经很清楚时。

### 为什么

- 阶段明显
- 需要显式推进流程
- 可能需要在复审时暂停
- 失败后不希望从头来过

### 最关键的学习点

“检索 + 总结” 还是高层原型；“多阶段研究流程” 已经更像图系统。

## 案例 2: 审批工作流 Agent

### 场景

系统接收任务、生成计划、等待审批、执行动作，并在拒绝时回到修订阶段。

### 当前适合 LangGraph 吗

非常适合。

### 为什么

- 审批是正式暂停点
- 拒绝与通过构成显式分支
- 需要恢复继续执行
- 控制流比高层原型复杂得多

## 案例 3: FAQ Agent 升级版

### 场景

FAQ Agent 除了答疑，还要持续跟踪用户学习进度、在关键节点暂停确认学习计划，并可在中途恢复。

### 当前适合 LangGraph 吗

如果真的进入阶段跟踪和暂停恢复，就开始适合。

### 为什么

- 已不只是单轮答疑
- 需要跨阶段状态
- 需要清晰的暂停与继续逻辑

### Stage 3 判断关键

不是“FAQ 这三个字”，而是它背后的任务结构已经变了。

## 案例 4: 日程协调工作流

### 场景

系统查看空闲时间、提出候选、等待确认、重新协调、最终发送邀请，并在失败时重新选择策略。

### 当前适合 LangGraph 吗

通常适合，尤其当协调过程涉及多个阶段与反复修正时。

### 为什么

- 存在多轮分支
- 存在正式暂停等待用户确认
- 失败后希望回到中间步骤，而不是重来

## 案例 5: 企业审批与工单流转

### 场景

系统接收工单、判断类型、通知不同角色、等待审批或补资料、失败时退回到不同阶段。

### 当前适合 LangGraph 吗

非常适合。

### 为什么

- 状态明确
- 分支明确
- 审批明确
- 恢复路径明确

这是非常典型的图系统场景。

## 案例 6: 文档问答 RAG Agent

### 场景

用户提问，系统检索资料并给出答案，必要时附来源。

### 当前适合 LangGraph 吗

通常不一定。

### 为什么

- 如果只是高层 RAG 原型，LangChain 往往足够
- 除非你已经引入了多阶段检索、复核、审批和恢复

### Stage 3 判断关键

不是因为“用了 RAG” 就需要 LangGraph，而是因为任务结构是否已经变成图。

## 案例 7: 多阶段研究工作流

### 场景

系统分阶段完成研究计划、资料收集、观点整理、总结输出，并允许用户在关键节点修改方向。

### 当前适合 LangGraph 吗

很适合。

### 为什么

- 阶段天然明确
- interrupt 合理且必要
- 恢复路径有业务意义
- 前端往往也需要看到阶段进展

## 案例 8: 内容策略修订系统

### 场景

系统先生成策略草案，再等待人工批注，然后修改，再次审批，最终形成可交付版本。

### 当前适合 LangGraph 吗

适合，尤其当修订循环成为正式流程时。

### 为什么

- 修订不再是简单再问一次
- 而是图中的回路
- 每次修订后都可能进入不同分支

## 一张总表看差异

| 案例 | 当前是否适合 LangGraph | 关键原因 | 最关键的图能力 |
| --- | --- | --- | --- |
| 单工具研究助理升级版 | 开始适合 | 阶段和复审出现 | state |
| 审批工作流 Agent | 是 | 审批与恢复是核心 | interrupt + recovery |
| FAQ Agent 升级版 | 条件适合 | 学习计划阶段化 | state + interrupt |
| 日程协调工作流 | 是 | 多阶段确认与回退 | branching |
| 企业审批与工单流转 | 是 | 正式流程系统 | graph orchestration |
| 文档问答 RAG Agent | 通常不一定 | 高层原型仍足够 | 无需强上图 |
| 多阶段研究工作流 | 是 | 分阶段研究与修正 | state + persistence |
| 内容策略修订系统 | 是 | 修订循环正式化 | loop + approval |

## 使用建议

读完这个文件后，建议你选 2 到 3 个案例，回答下面这些问题:

1. 这个项目最关键的 state 是什么。
2. 这个项目最该放 interrupt 的位置在哪里。
3. 这个项目如果失败，最应该从哪一步恢复。
4. 为什么这里不能只靠高层 LangChain 原型继续做下去。

如果你能稳定回答这些问题，`Stage 3` 的图系统判断就开始成形了。
