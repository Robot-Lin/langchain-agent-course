# 05 Challenges

这一部分开始要求学习者不只是“看懂 LangChain 概念”，而是主动完成高层原型设计、输出结构设计、边界判断和复盘。

建议按顺序做，不要跳题。因为 `Stage 2` 真正训练的是一个渐进过程:

- 先判断
- 再结构化
- 再减法
- 再复盘
- 最后进入独立项目

## Challenge 1: 高层边界判断练习

### 目标

训练你判断: 哪些设计还适合继续留在 `LangChain`，哪些已经开始需要 `LangGraph` 思维。

### 任务

请判断下面哪些项目更适合继续停留在 LangChain 高层原型，哪些已经明显开始逼近 LangGraph。

### 题目 A

单工具研究助理，输出结构化摘要。

### 题目 B

系统需要多步骤审批、失败恢复和长任务暂停。

### 题目 C

结构化抽取 Agent，输入一段文本，输出固定 JSON。

### 题目 D

多阶段报告生成流程，需要反复回到前一步修订。

### 你至少要回答

- 当前更适合 LangChain 还是 LangGraph
- 为什么
- 最关键的判断依据是什么

### 合格标准

- 不只给结论
- 能说出主路径复杂度、状态复杂度、恢复需求等原因
- 不把“能做”误当成“适合继续做在这一层”

## Challenge 2: structured output 设计练习

### 目标

训练你把输出设计当成系统能力，而不是格式美化。

### 任务

请选择一个场景，为它设计 structured output:

- 面试助手
- 研究助理
- FAQ 助手升级版

### 你至少要写清

- 字段名
- 每个字段为什么需要
- 哪些字段给人看
- 哪些字段给系统继续消费
- 哪些字段用于表达不确定性或来源

### 合格标准

- schema 不是只有一个 summary
- 字段和任务目标强相关
- 能看出后续系统如何消费这些结构

## Challenge 3: 工具边界练习

### 目标

训练你从“工具越多越强”退回到“工具是否必要”。

### 任务

你现在有下面 4 个工具候选:

- 搜索
- 发邮件
- 数据库读写
- 文件系统

请为一个“研究助理”只保留必要工具，并说明为什么删掉其他工具。

### 你至少要回答

- 最终保留什么工具
- 为什么这个工具就足够验证主路径
- 其他工具为什么现在不需要

### 合格标准

- 决定有明确理由
- 不是为了“以后可能会用到”而保留
- 能说明工具和任务目标的对应关系

## Challenge 4: middleware 设计练习

### 目标

训练你开始把控制逻辑从 prompt 中分层出去。

### 任务

请写出一个高层 Agent 可以放进 middleware 的 3 类逻辑，并说明为什么不建议把它们都塞进 prompt。

### 建议思考方向

- 人工审核
- 工具限制
- 风险拦截
- 行为守护规则

### 合格标准

- 能说出 middleware 的作用不是“多一层技术”，而是“让控制逻辑更清楚”
- 能解释 prompt 与 middleware 的职责差异

## Challenge 5: 原型失败复盘练习

### 目标

训练你把高层原型失败看成设计问题，而不是只怪模型。

### 任务

假设一个 LangChain 原型出现了下面这些问题，请判断它们更像是哪类设计问题:

- 找到了资料，但输出很难继续加工
- 能回答，但经常跳过澄清步骤
- 工具结果被直接照抄，没有整理
- 系统开始出现很多例外分支，prompt 越写越长

### 你至少要回答

- 这更像主任务问题、工具问题、输出问题，还是控制逻辑问题
- 为什么会发生
- 系统层该怎么改

### 合格标准

- 改进思路不是只说“换模型试试”
- 能区分 structured output 问题和控制逻辑问题
- 能看出“开始逼近 LangGraph”的信号

## Challenge 6: Stage 1 升级练习

### 目标

训练你把 `Stage 1` 的系统设计推进成 `Stage 2` 的高层原型设计。

### 任务

请把你在 `Stage 1` 设计过的一个 Agent 升级成 LangChain 高层原型，并至少补上:

- 最小工具清单
- structured output
- middleware / 控制逻辑
- 为什么当前仍适合 LangChain

### 合格标准

- 能看出从 Stage 1 到 Stage 2 的升级
- 不是简单把旧方案改个名字
- 新增内容真正围绕高层原型能力展开

## Challenge 7: 综合小作业

### 目标

把前面练习过的能力拼起来，完成一个完整的高层原型说明。

### 任务

任选一个你熟悉的场景，写出一份 `220-450` 字的 LangChain 高层 Agent 原型说明，并附带:

- 最小工具清单
- structured output 设计
- middleware 规则草案
- 至少 4 个失败点
- 一段“为什么当前仍适合 LangChain”的边界说明

### 合格标准

- 结构完整
- 输出设计清楚
- 工具克制
- 控制逻辑有层次
- 边界判断清楚

## 自评 Rubric

做完这一章后，你可以按下面 5 个维度给自己打分，每项 `1-5` 分:

- 高层理解: 我能否解释 LangChain 为什么适合作为高层入口
- 输出意识: 我能否优先设计 structured output
- 工具边界: 我能否控制工具数量并说明理由
- 控制层次: 我能否分清 prompt 与 middleware 的职责
- 边界判断: 我能否判断项目何时该进入 LangGraph

### 粗略参考

- `20-25` 分: 可以进入课后项目或准备进入 Stage 3
- `15-19` 分: 建议再做一轮挑战题
- `14` 分及以下: 建议回看 `02-concepts.md` 和 `03-methods.md`

## 推荐学习闭环

每完成一个 challenge，都做一次小闭环:

1. 写下你的答案。
2. 让 Codex 只做评审。
3. 记录它指出的关键问题。
4. 用你自己的语言修正答案。
5. 总结这题帮你建立了什么高层原型判断标准。
