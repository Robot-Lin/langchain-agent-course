# 06 Review Checklist

这个文件的作用不是让你“打卡完成 Stage 2”，而是帮你判断: 你到底有没有真正掌握 LangChain 高层原型思路，以及你是否已经准备好进入 `Stage 3`。

## 这一章怎么用

建议你分 3 次使用这个清单:

1. 学完概念和方法后，做第一次自测。
2. 做完引导项目后，检查高层原型结构是否清楚。
3. 做完挑战题和课后项目后，做正式放行判断。

真正有效的用法不是机械打勾，而是要求自己: 每一项都能举出一个自己的例子，或者一个明确反例。

## 第一部分: 高层概念理解自测

如果下面任意 3 项做不到，建议不要急着进入 `Stage 3`。

- 我能解释 LangChain 为什么是高层 Agent 开发入口。
- 我能解释 LangChain 和 LangGraph 的上下层关系。
- 我能说明 tools 在当前项目里为什么必要。
- 我能设计一套结构清楚的 structured output。
- 我能解释 middleware 在系统中的作用。
- 我能说明高层 human-in-the-loop 为什么有价值。
- 我能做一个最小 LangChain 原型设计。
- 我能指出什么时候高层原型开始不够用了。

### 合格信号

- 你不是只会说“LangChain 比较方便”
- 你能解释它帮你抽象了什么
- 你能说出它与 LangGraph 的边界区别

### 危险信号

如果你还有这些表现，建议回看概念部分:

- 把 LangChain 当成“写代码更少的模型调用器”
- 不知道为什么 structured output 比 prompt 文案更关键
- 说不出 middleware 与 prompt 的职责差异

## 第二部分: 高层原型结构自测

请对你自己的一个方案做检查。如果下面这些问题答不清，说明你还没有真正掌握高层原型思路。

- 我是否定义了清楚的主任务，而不是一串功能。
- 我是否只保留了最小必要工具。
- 我是否优先设计了 structured output。
- 我是否说明了哪些逻辑适合放在 middleware。
- 我是否说明了这里需不需要 short-term memory。
- 我是否说明了哪里需要高层 human-in-the-loop。
- 我是否说明了什么输出才算“可被人和系统使用”。
- 我是否说明了当前为什么仍适合 LangChain。

### 合格信号

你的方案读起来应该有这种感觉:

- 任务聚焦
- 工具克制
- 输出清楚
- 控制层次分明
- 边界意识明确

### 危险信号

如果你的方案里出现这些情况，说明还在过度设计:

- 还没跑通主路径就上很多工具
- structured output 非常模糊
- 所有控制逻辑都混进 prompt
- 明明已经是复杂流程，却硬说“LangChain 也能凑合”

## 第三部分: 风险与失败自测

一个 `Stage 2` 合格学习者，不只会写“能工作的原型”，还应该能先看到高层原型最容易失败在哪。

请检查自己是否能做到:

- 至少说出 4 个失败点
- 能区分主任务问题、工具问题、输出问题、控制逻辑问题
- 能指出至少 2 个需要审核或人工确认的场景
- 能说明哪些问题说明项目已经开始逼近 LangGraph

### 合格信号

- 失败点具体，不抽象
- 改进思路针对结构，而不是只说“换模型”
- 能看到输出与控制层失败，而不只盯工具调用

### 危险信号

- 所有失败都归因给模型
- 完全没有写“何时该升级到 LangGraph”
- 没有设计任何审核场景

## 第四部分: 表达与复述自测

进入 `Stage 3` 前，你应该不看稿也能把一个 LangChain 高层原型讲清楚。

请检查自己是否能脱稿回答:

- LangChain 为什么适合作为高层入口
- tools、structured output、middleware 分别解决什么问题
- 为什么工具不该越多越好
- 为什么 high-level human-in-the-loop 很重要
- 什么迹象说明该进入 LangGraph

### 合格信号

- 回答里有系统结构意识
- 有你自己的项目例子
- 能解释“为什么现在留在 LangChain 是合理的”

### 危险信号

- 只能讲概念，讲不出项目判断
- 一离开术语就不会说了
- 只能说“能做”，讲不出“是否适合继续做在这一层”

## 阶段答辩题

下面这组问题可以作为 `Stage 2` 的正式放行答辩题。

### 必答题

1. 为什么 structured output 是 Agent 系统能力，而不只是格式美化。
2. 为什么工具不应该越多越好。
3. 为什么 middleware 比全部塞进 prompt 更适合放某些控制逻辑。
4. 什么情况下 LangChain 高层原型已经开始不够用。

### 进阶题

1. 为什么高层原型也应该尽早考虑 human-in-the-loop。
2. 为什么一个“能跑”的 LangChain 原型仍然可能不适合上线。
3. 如果你要把 Stage 1 的 FAQ Agent 升级到 Stage 2，你会先补什么。

### 合格标准

- 每题都能结合项目场景回答
- 每题都能提到边界或失败风险
- 不只是会复述定义，而是真的会判断

## 项目产物检查

进入下一阶段前，建议你至少完成下面这些产物:

- 1 份 LangChain 高层原型设计说明
- 1 个跟学项目拆解结果
- 1 组挑战题答案
- 1 次基于标准的自我答辩或 Codex 评审
- 1 份课后项目方案

如果这些产物都没有，说明你更像“读过 Stage 2”，而不是“掌握了 Stage 2”。

## 放行标准

只有当你已经能意识到下面这些事时，`Stage 3` 才会真正有效:

- 高层抽象的价值是什么
- structured output 为什么优先级很高
- 控制逻辑为什么要适当分层
- 高层原型的边界在哪里
- 什么时候继续堆高层封装已经不划算

更具体一点说，你至少要能做到:

- 不乱加工具
- 不把输出设计留到最后
- 不把所有控制逻辑都写进 prompt
- 不忽略审核与风险边界
- 能说清项目何时该进入 LangGraph

在进入 `Stage 3` 之前，先问自己一句:

如果你能明确回答后者，说明 `Stage 2` 的关键价值已经开始内化。
