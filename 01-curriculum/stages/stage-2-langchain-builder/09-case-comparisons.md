# 09 Case Comparisons

这个文件用于训练 `Stage 2` 的关键能力: 不是一看到框架就开搭，而是先判断一个项目当前是否真的适合停留在 `LangChain` 高层。

建议你一边读，一边回答:

- 为什么这个场景适合 LangChain
- 为什么它暂时不需要 LangGraph
- 如果继续复杂化，哪个点最先逼近 Stage 3

## 案例 1: FAQ Agent 升级版

### 场景

你已经有一个课程 FAQ Agent，现在希望它能读取课程资料，给用户推荐学习入口，并输出结构化推荐结果。

### 当前适合 LangChain 吗

适合。

### 为什么

- 主路径清楚
- 工具需求轻量
- 输出结构可以先明确设计
- 不需要复杂状态机

### 什么时候会开始逼近 LangGraph

- 需要多轮跟踪学习计划
- 需要审批和恢复成为主流程能力
- 需要复杂阶段推进逻辑

## 案例 2: 单工具研究助理

### 场景

用户提出研究问题，系统使用一个检索工具获取资料，输出结构化研究摘要。

### 当前适合 LangChain 吗

很适合。

### 为什么

- 单工具主路径清楚
- structured output 价值明显
- 可以先用高层 agent 快速原型化

### 最关键的学习点

- 工具最小化
- 输出结构优先
- 边界清楚比“更智能”重要

## 案例 3: 结构化面试 Agent

### 场景

系统接收岗位描述和候选人回答，动态追问并输出结构化面试结论。

### 当前适合 LangChain 吗

通常适合。

### 为什么

- 虽然有动态追问，但还属于高层可控原型
- structured output 非常关键
- 基础 human-in-the-loop 也能在高层接入

### 什么时候会开始逼近 LangGraph

- 多轮评估流程明显分阶段
- 面试进程需要恢复和中断
- 需要审批、转交和复杂状态跟踪

## 案例 4: 文档问答 RAG Agent

### 场景

系统从外部知识源检索内容，回答用户问题，并输出带来源的结构化结果。

### 当前适合 LangChain 吗

适合，而且是很典型的 Stage 2 场景。

### 为什么

- 高层 RAG agent 原型非常契合 LangChain
- 重点在检索接入、输出结构和结果边界
- 不一定需要复杂图编排

### Stage 2 最该学什么

- 资料接入后，输出边界更重要
- 检索不是“加个工具就完了”，而是改变了回答可靠性的设计要求

## 案例 5: 企业审批助手

### 场景

系统需要判断提交内容、查询信息、通知相关人，并在多个节点等待不同角色审批。

### 当前适合 LangChain 吗

第一版可以做高层原型，但很快会逼近 LangGraph。

### 为什么

- 如果只是单次分析和建议，LangChain 足够
- 如果审批流本身成为主流程核心，就会需要图编排和恢复机制

### Stage 2 判断关键

先分清你要做的是:

- 一个“给建议的高层原型”
- 还是一个“真正驱动审批流程的系统”

很多项目真正难的不是模型回答，而是流程本身。

## 案例 6: 日程协调 Agent

### 场景

系统查看空闲时间、给出候选时间、让用户确认、再发出邀请。

### 当前适合 LangChain 吗

第一版通常适合。

### 为什么

- 工具数量有限
- 控制逻辑还不算复杂
- human-in-the-loop 很清楚

### 什么时候会开始逼近 LangGraph

- 需要跨多方持续协调
- 需要处理中断与恢复
- 需要长生命周期状态跟踪

## 案例 7: 内容策略 Agent

### 场景

系统分析目标受众、制定内容方向、生成主题建议，并持续根据反馈修订计划。

### 当前适合 LangChain 吗

可以先做高层原型，但复杂度上升会比较快。

### 为什么

- 第一版可以先聚焦在“给结构化策略建议”
- 一旦开始多阶段规划、持续修订、上下文压缩，复杂度会明显增长

### 对课程的意义

这是一个很好的“Stage 2 到 Stage 4 过渡型案例”。

## 案例 8: 多步骤研究工作台

### 场景

系统需要分阶段完成问题澄清、检索、交叉验证、摘要、复审、最终交付。

### 当前适合 LangChain 吗

只适合非常轻的第一版；如果阶段性流程明确，很快会更适合 LangGraph。

### 为什么

- 主流程已经明显不是单条直线
- 中间状态和恢复能力会变重要
- 评审点和阶段边界更像图系统而不是纯高层原型

## 一张总表看差异

| 案例 | 当前是否适合 LangChain | 关键原因 | 最可能逼近 Stage 3 的点 |
| --- | --- | --- | --- |
| FAQ Agent 升级版 | 是 | 轻量资料接入与推荐 | 多轮学习计划状态 |
| 单工具研究助理 | 是 | 单工具 + 结构化输出 | 工具和流程增多 |
| 结构化面试 Agent | 是 | 高层追问与结构化结果 | 面试流程分阶段 |
| 文档问答 RAG Agent | 是 | 高层 RAG 原型典型场景 | 检索与验证流程复杂化 |
| 企业审批助手 | 第一版可行 | 分析建议可高层实现 | 审批流成为主系统 |
| 日程协调 Agent | 第一版可行 | 工具少且确认点清楚 | 长生命周期协调 |
| 内容策略 Agent | 第一版可行 | 先做结构化策略输出 | 长上下文规划与修订 |
| 多步骤研究工作台 | 轻版勉强可行 | 主流程很快复杂化 | 阶段图与恢复机制 |

## 使用建议

读完这个文件后，建议你选 2 到 3 个案例，回答下面这些问题:

1. 这个项目第一版最少应该保留哪些工具。
2. 这个项目最先应该明确的 structured output 是什么。
3. 这个项目现在为什么还适合留在 LangChain。
4. 它最先会在哪个点逼近 LangGraph。

如果你能稳定回答这些问题，`Stage 2` 的边界判断就开始成形了。
