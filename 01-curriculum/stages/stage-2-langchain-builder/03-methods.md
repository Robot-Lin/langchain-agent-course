# 03 Methods

本文件关注的是 Stage 2 的做法，而不是概念定义。

## 方法一: 高层原型四步法

面对一个 LangChain 项目，先做四步:

1. 定义唯一主任务
2. 定义必要工具
3. 定义结构化输出
4. 定义基本控制逻辑

这样做的原因是:

- 主任务决定系统边界
- 工具决定外部能力
- 输出决定系统可验证性
- 控制逻辑决定系统可控性

## 方法二: 先设计输出，再设计提示词

Stage 2 最重要的习惯之一，是先问:

- 我最终要什么结构的结果
- 谁要消费这个结果
- 这个结果是给人看，还是给系统继续处理

如果输出结构不清楚，提示词再漂亮也只是“表面流畅”。

## 方法三: 工具最小化

不要一上来就给 Agent 很多工具。

推荐顺序:

1. 零工具版本
2. 单工具版本
3. 必要时再增加第二个工具

只有当学习者能说明“新增这个工具会让系统完成什么原本做不到的事情”，才应该继续加。

## 方法四: Middleware 外挂法

不要把所有规则都写进 prompt。

适合放进 middleware 的内容包括:

- 人工审核
- 特定工具限制
- 风险拦截
- 简单策略控制

这会让系统更易解释，也更利于后续迁移到更复杂框架。

## 方法五: 高层失败复盘法

Stage 2 的失败通常来自四类问题:

- 主任务定义不清
- 工具设计不清
- 输出结构不清
- 控制逻辑缺失

每次复盘都建议按这四类去查。

## 方法六: 何时停止留在 LangChain

出现下面情况时，要开始准备进入 Stage 3:

- 你需要明显的多分支流程
- 你需要长任务恢复
- 你需要复杂状态控制
- 你需要中断、批准和恢复成为系统主能力

这时再继续“硬塞”在高层原型里，维护成本会快速上升。

## Stage 2 方法目标

真正掌握本阶段，不是“会调用 LangChain”，而是能把一个合理的 Agent 设计推进成结构清晰、输出可靠、可演示的高层原型。
