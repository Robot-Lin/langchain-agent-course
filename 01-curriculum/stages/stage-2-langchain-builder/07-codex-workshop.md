# 07 Codex Workshop

这个文件不是“给 Codex 的说明”，而是“给学习者的 Stage 2 协作手册”。它的目的不是让 Codex 更快帮你做完原型，而是让你在和 Codex 的配合里，真正学会高层原型设计。

## Stage 2 与 Codex 配合的核心原则

在这一阶段，Codex 最适合扮演这些角色:

- 高层概念解释者
- 原型设计教练
- 工具边界审查员
- structured output 评审员
- LangChain / LangGraph 分界提醒员

在这一阶段，Codex 最不适合扮演的角色是:

- 直接替你完成整套原型方案的人
- 一上来就给你“全工具全能力版本”的人
- 用复杂度掩盖结构问题的人

一句话原则:

> 在 Stage 2，Codex 应该帮你形成高层原型判断，而不是替你跳过高层原型设计。

## 使用前先告诉 Codex 什么

每次开始一个会话前，建议你至少告诉 Codex 这 5 件事:

1. 我现在处于 `Stage 2`。
2. 我的目标是学习高层原型设计，不是直接拿完整实现。
3. 请先问我问题，再帮我评审。
4. 不要一次性展开成复杂系统设计。
5. 如果你觉得已经逼近 LangGraph，请明确提醒我。

一个通用开场模板:

```text
我现在在学习 LangChain Agent Course 的 Stage 2。
这一阶段我想训练的是 LangChain 高层 Agent 原型设计，不是直接拿完整实现。
请你先做导师和评审，不要直接给我完整答案。
每次只推进一步，先问我问题，再指出我的结构问题。
如果你判断这个项目已经开始逼近 LangGraph，请明确告诉我原因。
```

## 模式一: 高层概念讲解模式

适用场景:

- 你刚读完概念，但还不真正理解 LangChain 的价值
- 你知道 tools 和 structured output 的定义，但不知道为什么它们在这一章重要
- 你还说不清 LangChain 与 LangGraph 的边界

推荐提问:

```text
请用课程学习者能听懂的方式解释 LangChain 为什么适合作为高层 Agent 原型入口。
不要只讲定义，要结合真实项目说明。
```

```text
请结合一个真实场景解释 structured output 的价值。
重点讲为什么它比“漂亮回答”更重要。
```

```text
请比较 prompt 控制和 middleware 控制的差别。
重点讲系统层面的影响，不要只讲 API。
```

### 正确用法

- 让 Codex 解释后，自己先复述
- 再让 Codex 指出你复述中的漏洞

### 错误用法

- 看完解释就继续下一章
- 不做复述，误以为“看懂了就算掌握”

## 模式二: 高层原型拆解模式

适用场景:

- 你已经有一个 Agent 想法，想推进成 LangChain 原型
- 你不确定主任务、工具、输出、控制逻辑该怎么定义
- 你想练高层原型四步法

推荐提问:

```text
我想把这个 Stage 1 方案升级成 LangChain 高层原型。
请先不要写代码，先按主任务、工具、structured output、控制逻辑四步带我拆解。
每一步先提问，不要直接帮我填答案。
```

```text
请先判断这个项目现在是否适合留在 LangChain。
如果不适合，请明确告诉我最主要的原因。
```

### 正确用法

- 先自己答
- 再让 Codex 审查
- 最后自己改

### 错误用法

- 直接让 Codex 生成完整原型文档
- 把 Codex 生成的结构当成自己的理解

## 模式三: structured output 设计模式

适用场景:

- 你总是习惯先写 prompt，忘了先设计输出
- 你不确定字段如何划分
- 你不知道哪些字段是给系统消费的

推荐提问:

```text
请帮我审查这个 Agent 的 structured output 设计。
不要直接替我重写，先指出哪些字段太模糊、哪些字段缺失、哪些字段无法被后续系统消费。
```

```text
请帮我区分这里哪些字段主要给人看，哪些字段主要给系统后续使用。
```

### 正确用法

- 先提交自己的 schema 草案
- 再让 Codex 从结构角度批评

### 错误用法

- 一开始就让 Codex 代你设计整套 schema

## 模式四: 工具与边界评审模式

适用场景:

- 你不确定工具是不是加多了
- 你担心自己只是为了“显得高级”在加功能
- 你想知道项目是否开始逼近 LangGraph

推荐提问:

```text
请只从工具边界角度审查我的方案。
告诉我哪个工具最值得删，哪个工具是真正支撑主路径的。
不要帮我加新工具。
```

```text
请站在架构评审角度，判断这个项目现在还适不适合继续留在 LangChain 高层。
请明确指出最关键的判断信号。
```

### 正确用法

- 接受删减
- 主动要求 Codex 指出“最不像高层原型”的部分

### 错误用法

- 每次被指出问题，就继续加功能证明自己

## 模式五: 项目带学模式

适用场景:

- 你在做 [04-guided-project.md](./04-guided-project.md)
- 你在做 [05-challenges.md](./05-challenges.md)
- 你准备进入课后项目

推荐提问:

```text
请按 Stage 2 的知识点顺序带我完成单工具研究助理原型。
每次只推进一步，并先告诉我这一步对应哪个概念和方法。
不要一次给完整答案。
```

```text
我现在做的是 Stage 2 课后项目。
请不要代做，只按主任务、工具、structured output、控制逻辑和边界做导师和评审。
```

### 正确用法

- 用 Codex 做带学教练
- 每一步都先解释为什么
- 保留你自己的初版答案和修改痕迹

### 错误用法

- 跳过自己的回答，直接复制 Codex 的方案

## 模式六: 模拟答辩模式

适用场景:

- 你准备做 Stage 2 放行检查
- 你想测试自己是否已经能清楚讲出高层原型思路
- 你希望进入 Stage 3 前做一次边界压力测试

推荐提问:

```text
你来当导师，逐题提问我 Stage 2 的答辩题。
不要先给答案，等我回答完再指出哪里不清楚。
```

```text
请重点追问我，为什么这个项目还适合 LangChain，以及它何时会开始需要 LangGraph。
```

### 正确用法

- 先脱稿回答
- 再让 Codex 追问
- 最后自己写复盘

### 错误用法

- 让 Codex 先给标准答案再背

## 推荐学习动作

一个很稳的对练闭环是:

1. 先让 Codex 解释。
2. 再自己复述。
3. 再让 Codex 挑错。
4. 再修正你的表达。
5. 最后让 Codex 模拟评审或答辩。

这个顺序很重要，因为它会逼你从“看懂高层框架”进入“能判断高层原型是否合理”。

## 什么时候不要让 Codex 直接回答

在下面这些场景里，最好不要让 Codex 直接给完整答案:

- 你正在做 challenge
- 你正在做课后项目
- 你准备做答辩
- 你还没有自己写过第一版高层原型方案

更好的做法是:

- 先写你的版本
- 再让 Codex 做评审
- 最后自己再改一版

## 一个完整的 Stage 2 会话模板

你可以直接复制下面这段开始一轮高质量学习:

```text
我现在在学习 LangChain Agent Course 的 Stage 2。
我希望你扮演导师和评审，而不是直接代做。
请先根据我给你的场景，判断它当前是否适合继续留在 LangChain 高层。
如果适合，再按主任务、工具、structured output、控制逻辑这几个维度，一步一步带我拆解。
每次只推进一步。
先问我问题，再指出我回答里的结构问题。
不要直接进入完整实现，也不要一次给完整答案。
如果你觉得这个项目已经开始逼近 LangGraph，请明确提醒我原因。
```

## Stage 2 结束前，你应该达到的协作状态

如果这一章学得比较扎实，到了最后你和 Codex 的关系应该是:

- 你负责提出高层原型草案
- Codex 负责挑战你的工具和输出设计
- 你负责修正控制逻辑和边界
- Codex 负责指出是否开始逼近 LangGraph
- 你最终能不依赖 Codex，也把一个 LangChain 高层原型讲清楚

这才说明 Codex 真正起到了“放大学习效率”的作用，而不是替你绕过了高层原型思考过程。
