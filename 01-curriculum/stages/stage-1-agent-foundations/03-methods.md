# 03 Methods

这一章不再讲“概念是什么”，而是讲“拿到一个想法以后，应该怎么下手”。

## 方法一: 先判断这是不是一个 Agent 问题

在开始设计前，先做一个快速判断。你可以连续问自己 6 个问题:

1. 这个系统除了回答，还需要真的“做事”吗。
2. 它需要访问外部世界吗，例如文件、搜索、数据库、API。
3. 它需要记住任务进度或中间结果吗。
4. 它需要根据中间结果改变下一步动作吗。
5. 它需要和人来回确认、审批或澄清吗。
6. 如果不用 Agent，普通规则逻辑是否已经足够。

如果前 5 个问题里大多数答案是否定的，而第 6 个问题是肯定的，那通常说明:

> 这还不是一个优先应该做成 Agent 的问题。

这个判断非常重要，因为它能帮你避免过度设计。

## 方法二: 需求拆解五问

如果初步判断确实需要 Agent，再继续做需求拆解。这里建议用 `五问法`。

1. 这个系统为谁服务。
2. 它最核心的单一任务是什么。
3. 它必须连接哪些外部能力。
4. 哪些地方需要跨轮延续上下文。
5. 哪些动作必须让人确认。

这五问的作用是把模糊的“我想做个 Agent”压缩成清晰的结构。

如果这五问回答不清，通常不建议直接进入实现。

## 方法三: 最小 Agent 设计画布

在真正开工前，先填一张最小 Agent 画布。

### 1. 用户

- 谁在使用它
- 用户最常见的触发场景是什么
- 用户在什么情况下会觉得它“有帮助”

### 2. 目标

- 这个系统唯一最重要的目标是什么
- 哪些“看起来不错”的附加功能，现在先不做

### 3. 输入

- 用户会给什么输入
- 系统还能接到什么上下文或外部输入

### 4. 外部能力

- 需要哪些工具
- 每个工具为什么必要
- 哪些工具故意不接

### 5. 状态

- 这次任务里，系统必须持续保留什么
- 如果忘掉这些信息，会发生什么问题

### 6. 记忆

- 哪些信息只在当前任务有效
- 哪些信息跨会话也值得保留
- 哪些东西其实不该放进记忆，而应该放进知识库

### 7. 输出

- 输出给谁看
- 输出是自然语言、结构化字段，还是可执行动作
- 输出怎么判断是“可用的”

### 8. 风险

- 哪些地方最可能出错
- 哪些地方一旦出错后果最大

### 9. 人工介入

- 哪些地方必须暂停
- 暂停后需要什么形式的确认

你可以把这张画布理解成 Agent 项目的 `产品定义 + 系统边界 + 风险草图`。

## 方法四: 先做减法，再做加法

新手常见的起手式是:

- 多工具
- 多步骤规划
- 多 Agent
- 长期记忆
- 前后端一起上

`Stage 1` 要刻意反过来做。

第一版只保留:

- 一个核心用户
- 一个核心任务
- 一条主路径
- 最少必要工具
- 最少必要状态

如果第一版连边界都讲不清，就不要急着加复杂度。

成熟的系统不是一开始就复杂，而是:

> 先有一个清楚的简单版本，再有理由地增加复杂度。

## 方法五: 失败前置设计

不要等系统做完才想“会不会出错”。在项目启动前，就先写失败点。

建议至少从 4 类失败开始:

### 1. 理解失败

- 用户目标太模糊
- 系统误判意图
- 系统问错澄清问题

### 2. 工具失败

- 工具拿不到结果
- 工具结果不完整
- 系统调用了不该调用的工具

### 3. 流程失败

- 漏掉关键步骤
- 提前结束任务
- 卡在某个中间状态回不来

### 4. 交互失败

- 输出太笼统
- 用户不知道系统在做什么
- 该暂停时没有暂停

一个很有帮助的问题是:

> 如果这个系统失败了，用户最可能先抱怨什么。

这个问题通常能帮你看到真正的系统薄弱点。

## 方法六: 审批门设计

给系统设置“不能自动越界”的门。

这类门通常出现在:

- 对外发送消息前
- 花钱前
- 删除前
- 发布前
- 修改正式数据前

你可以把审批门理解成系统里的刹车装置。

一个简单模板:

- 什么动作不能自动执行
- 谁来确认
- 用户看到什么信息后做确认
- 拒绝后系统怎么退回

如果这四个点说不清，审批设计通常还不够完整。

## 方法七: 用自然语言写架构，而不是先画技术图

在没写任何代码前，先要求自己能用一段自然语言把系统讲清楚。

推荐模板:

> 这个 Agent 面向谁，解决什么问题。它接收哪些输入，调用哪些外部能力，保留哪些状态，是否需要记忆，在什么地方需要人工确认，最后输出什么结果，最容易失败的地方在哪里。

如果你能把这段话说清楚，后面再进框架实现会轻松很多。

如果这段话说不顺，通常说明以下至少有一项还没想清楚:

- 用户不清楚
- 目标太散
- 工具边界不清
- 状态和记忆混了
- 失败路径没设计

## 方法八: 用 Codex 做“设计评审”，而不是直接代做

在 `Stage 1`，最推荐的 AI 协作方式不是让 Codex 直接给方案，而是让它扮演这几种角色:

- 需求拆解教练
- 边界审查员
- 风险评审员
- 方案压缩助手

推荐提问方式:

- 帮我判断这个需求是否真的需要 Agent。
- 不要直接给答案，先问我 5 个澄清问题。
- 站在审查角度指出我这个方案里最容易过度设计的部分。
- 帮我区分这里哪些属于状态，哪些属于记忆，哪些属于知识库。

这样学出来的能力会更稳，因为你是在训练判断，而不是抄系统答案。

## Stage 1 的方法目标

真正掌握本阶段，不是能背出概念，而是能把一个模糊想法压缩成一个合理、可解释、可迭代的最小系统。
