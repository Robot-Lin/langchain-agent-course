# 03 Methods

本文件不是讲概念，而是教学习者如何下手。

## 方法一: 需求拆解五问

面对任何一个想做的 Agent，先问五个问题:

1. 这个系统为谁服务
2. 它的核心任务是什么
3. 它必须连接哪些外部能力
4. 哪些地方需要连续上下文
5. 哪些动作必须人工确认

如果这五问答不清，大概率还不适合开始实现。

## 方法二: 最小 Agent 画布

给每个项目都先填这张画布。

### 用户

- 谁在使用它
- 用户最常见的触发方式是什么

### 目标

- 它最核心的唯一目标是什么

### 输入

- 用户提供什么
- 系统还能收到什么外部输入

### 外部能力

- 需要哪些工具
- 需要哪些数据源

### 状态

- 执行过程要记住什么

### 记忆

- 哪些信息只在当前任务有效
- 哪些信息需要长期保留

### 输出

- 输出给谁看
- 输出格式是否可消费

### 风险

- 哪些地方最可能出错
- 哪些地方后果最大

### 人工介入

- 在什么地方需要暂停
- 需要什么形式的确认

## 方法三: 先减法，再加法

新手很容易一开始就想做:

- 多工具
- 多轮规划
- 多代理
- 前后端全都有

Stage 1 的正确做法是:

- 只保留一个主任务
- 只保留一条主路径
- 只保留一个用户类型
- 只保留最必要的工具

先让结构清楚，再加复杂度。

## 方法四: 失败前置设计

在项目启动前，就要先写失败点，而不是做好后才补。

建议至少写这四类:

- 理解失败
- 工具失败
- 流程失败
- 交互失败

示例:

- 用户问题太模糊，系统误解目标
- 检索结果不完整，系统仍给出过度自信结论
- 工具调用后没有校验结果
- 系统准备发送邮件，却没有让人确认

## 方法五: 审批门设计

给系统设置“不能自动越过”的门。

这类门通常出现在:

- 发消息前
- 扣费前
- 删除前
- 对外发布前
- 修改正式数据前

你可以把它理解成系统的刹车装置。

## 方法六: 自然语言架构说明

在没写任何代码前，学习者应该能用下面模板说清架构:

“这个 Agent 面向谁，解决什么问题。它接收哪些输入，调用哪些外部能力，保留哪些状态，在什么地方需要人工确认，最终输出什么结果，哪些地方最容易失败。”

如果这句话说不顺，说明设计还不够清楚。

## Stage 1 方法目标

真正掌握本阶段，不是能背出概念，而是能把一个模糊想法压缩成一个合理、可解释、可迭代的最小系统。
