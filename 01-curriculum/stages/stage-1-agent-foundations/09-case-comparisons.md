# 09 Case Comparisons

这个文件用于训练一项 Stage 1 的关键能力: 不急着实现，而是先判断。

建议阅读方式不是从头背到尾，而是逐个案例判断:

- 它是不是 Agent 问题
- 为什么是，或者为什么不是
- 如果是，它最少需要什么能力
- 如果不是，更适合什么实现方式

## 案例 1: 固定格式周报润色器

### 需求

用户输入一段已经写好的周报，系统帮忙润色、压缩、调整成统一模板。

### 是否一定要做成 Agent

不一定。

### 为什么

- 主要是单轮文本加工
- 不需要外部动作
- 不需要状态管理
- 不需要审批流

### 更适合什么

- 一个普通 LLM 功能
- 或者一个结构化 prompt + 模板系统

### Stage 1 判断结论

这是一个很好用模型解决的问题，但未必是一个值得设计成 Agent 的问题。

## 案例 2: 课程 FAQ 助手

### 需求

用户询问 “我应该先学哪一章”“这节课适合我吗”“Stage 2 和 Stage 3 差别是什么”。系统需要结合课程目录给出推荐路径，并在用户目标模糊时先追问。

### 是否适合做成 Agent

可以做成一个轻量 Agent。

### 为什么

- 需要围绕用户目标给路径建议
- 可能需要读取课程资料
- 需要在目标模糊时做澄清
- 需要少量短期上下文

### 最少需要什么能力

- 文档读取或检索能力
- 短期对话状态
- 澄清策略

### 暂时不需要什么

- 长期记忆
- 多 Agent
- 高风险审批流

## 案例 3: 日程协调助手

### 需求

用户说 “帮我安排下周和产品经理开会”，系统需要看日历空闲时间、给出候选时间、征求确认，然后发出邀请。

### 是否适合做成 Agent

是。

### 为什么

- 需要访问外部工具
- 需要多步推进
- 需要在发出邀请前确认
- 需要记住当前正在协调哪场会议

### 最少需要什么能力

- 日历工具
- 当前任务状态
- 人工确认点

### 风险点

- 选错时间
- 发错人
- 未确认就发送邀请

## 案例 4: 报销审批助手

### 需求

系统接收报销申请，检查票据是否齐全，提取金额和类别，给出审核建议，并在关键节点把结果交给人审批。

### 是否适合做成 Agent

是，而且必须重视审批边界。

### 为什么

- 涉及结构化信息提取
- 涉及流程推进
- 涉及高风险业务动作
- 审批本身就是系统核心能力之一

### 最少需要什么能力

- 文档解析或结构化抽取
- 当前流程状态
- 人工审批门
- 审核日志或结果记录

### Stage 1 最该关注什么

不是“能不能自动批”，而是“哪里绝不能自动越界”。

## 案例 5: 深度研究助手

### 需求

用户给出一个开放问题，例如 “帮我研究东南亚 AI 教育产品机会”。系统需要搜集信息、归纳维度、形成结论，并可能继续深入某个方向。

### 是否适合做成 Agent

是，但复杂度明显更高。

### 为什么

- 需要大量工具使用
- 需要多步推进
- 需要中间结果汇总
- 需要根据发现动态调整研究路径

### 最少需要什么能力

- 搜索与资料读取
- 明确的研究状态
- 结构化输出
- 用户澄清和方向确认

### 它对应课程哪里

- Stage 1 只做边界判断
- Stage 4 才真正进入复杂任务型 Agent 设计

## 案例 6: 学习教练 Agent

### 需求

面向课程学习者，帮助用户明确目标、识别当前阶段、推荐下一步学习顺序，并在目标模糊时主动追问。

### 是否适合做成 Agent

适合，但第一版应该非常克制。

### 为什么

- 需要围绕目标做判断而不是只回答知识
- 需要少量短期状态
- 需要澄清和路径推荐

### 第一版最好不要加什么

- 长期记忆
- 复杂情绪分析
- 多 Agent 协作
- 自动代表用户执行外部动作

### Stage 1 学习价值

这个案例很适合做课后项目，因为它既像 Agent，又容易过度设计。

## 案例 7: 客服工单分流助手

### 需求

系统读取用户问题，把它归类到售前、售后、退款、Bug 反馈等类别，再给出建议去向，必要时交给人工客服。

### 是否适合做成 Agent

视边界而定。

### 如果只做分类

未必需要 Agent，更像结构化分类系统。

### 如果还要做什么

- 读取订单信息
- 查询账户状态
- 生成处理建议
- 识别高风险投诉并升级

那就开始更像 Agent。

### Stage 1 判断关键

先判断你要解决的是“分类问题”还是“任务推进问题”。很多项目其实只需要前者。

## 案例 8: 内容策略 Agent

### 需求

用户输入品牌目标，系统需要分析受众、生成内容主题、规划周内容日历，并在用户反馈后持续修订。

### 是否适合做成 Agent

适合，但更接近复杂任务系统。

### 为什么

- 目标模糊且开放
- 需要多步规划
- 需要上下文组织
- 需要多次与人对齐

### 它对应课程哪里

- Stage 1: 判断是否值得做成 Agent
- Stage 4: 学复杂任务规划与上下文工程

## 一张总表看差异

| 案例 | 是否值得做成 Agent | 关键原因 | Stage 1 最该学什么 |
| --- | --- | --- | --- |
| 周报润色器 | 通常不必 | 单轮文本加工 | 避免过度设计 |
| 课程 FAQ 助手 | 可以，且应轻量 | 需要资料读取和澄清 | 最小 Agent |
| 日程协调助手 | 是 | 工具 + 状态 + 确认 | 工具和状态 |
| 报销审批助手 | 是 | 流程 + 审批 + 风险 | 人工介入 |

## 使用建议

读完这个文件后，最推荐做的不是继续看更多概念，而是选 2 到 3 个案例，自己回答下面这些问题:

1. 如果只做第一版，我会删掉哪些能力。
2. 这个场景里最不该自动执行的动作是什么。
3. 我会不会把一个简单问题设计成了复杂 Agent。
4. 这个系统最少需要保留哪些状态。

如果你能稳定回答这些问题，`Stage 1` 的判断力就开始成形了。
