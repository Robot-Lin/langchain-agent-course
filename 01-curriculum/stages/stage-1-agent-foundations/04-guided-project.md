# 04 Guided Project

本阶段引导项目选择 `FAQ 助手`，因为它足够简单，但又能训练完整的 Agent 基础判断。

## 项目目标

设计一个能够回答课程常见问题、并在必要时引导用户继续学习的 FAQ 助手。

注意:

- 这一阶段不追求复杂实现
- 重点是练系统拆解
- 可以完全通过自然语言和 Codex 协作完成

## Step 1: 定义用户与目标

用户:

- 刚进入课程的新手

目标:

- 帮用户快速找到课程入口
- 回答课程相关常见问题
- 在用户目标不清时引导澄清

## Step 2: 明确输入与输出

输入:

- 用户问题
- 课程目录信息

输出:

- 问题回答
- 推荐阅读入口
- 必要时的澄清问题

## Step 3: 判断是否需要工具

先问:

- 它是否需要查询外部文档
- 它是否需要读取本地课程文件
- 它是否需要生成结构化推荐

结论:

- 这个系统至少需要“读取课程内容”的能力
- 但不需要高风险动作工具

## Step 4: 判断是否需要状态和记忆

需要最基本的短期状态:

- 当前用户在问哪类问题
- 上一轮给了什么建议

暂时不需要长期记忆:

- 因为这一阶段重点不是个性化教学，而是结构清晰

## Step 5: 设计失败路径

可能失败点:

- 问题太模糊
- 找不到对应课程内容
- 把课程建议说得太笼统
- 推荐路径和用户目标不匹配

## Step 6: 设计人工介入点

虽然这是低风险项目，但仍然可以设计“交给人”的场景:

- 当用户问题超出课程范围
- 当系统不确定推荐顺序
- 当用户目标描述过于模糊

## Step 7: 交付最小架构说明

学习者应该最终能说出:

“这是一个面向课程新手的 FAQ Agent。它接收用户问题和课程目录信息，帮助用户定位学习入口。系统需要读取课程内容，保留少量当前对话状态，不需要复杂长期记忆。它的主要风险是误解问题和给出错误推荐，因此在目标模糊时会先追问澄清。”

## 跟 Codex 的推荐对练

你可以直接对 Codex 说:

- 帮我把这个 FAQ Agent 拆成最小系统图
- 检查这个系统哪些部分其实不需要 Agent
- 站在评审角度指出我哪里设计过度

## 本项目真正训练的能力

- 目标压缩
- 边界判断
- 工具必要性判断
- 状态最小化
- 失败前置设计
