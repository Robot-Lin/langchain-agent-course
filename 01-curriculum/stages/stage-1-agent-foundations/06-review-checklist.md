# 06 Review Checklist

这个文件的作用不是让你“打卡”，而是帮你判断: 你到底有没有真正掌握 `Stage 1`，以及你是否已经准备好进入 `Stage 2`。

## 这一章怎么用

建议你分 3 次使用这个清单:

1. 学完概念和方法后，用它做第一次自测。
2. 做完引导项目后，用它检查结构漏洞。
3. 做完挑战题和课后项目后，用它做正式放行判断。

如果你只是看一遍就打勾，这个文件价值会很低。真正有用的方式是: 每一项都要求自己能举出一个例子，或者说出一个反例。

## 第一部分: 概念理解自测

如果下面任意 3 项做不到，建议不要急着进入 `Stage 2`。

- 我能用自己的话解释什么是 Agent，而且不会把它讲成“更强的提示词”。
- 我能区分 Agent、聊天机器人和固定流程逻辑。
- 我能说明工具在系统中的作用，而不是只说“工具让它更强”。
- 我能区分状态、短期记忆、长期记忆和知识库。
- 我能说出至少 3 类不同的失败，而不是只说“模型答错了”。
- 我能解释为什么有些动作必须让人确认。
- 我能把一个模糊需求压缩成最小系统，而不是一上来堆功能。

### 合格信号

你不是只会背术语，而是能:

- 举出自己的例子
- 举出反例
- 说明为什么某些场景不需要 Agent

### 危险信号

如果你还有这些表现，建议回看概念部分:

- 把 Agent 理解成“会自动回复的聊天机器人”
- 把知识库、记忆、状态写成一回事
- 一提“智能”就默认需要长期记忆和多工具

## 第二部分: 结构判断自测

请对你自己的一个方案做检查。如果下面这些问题答不清，说明结构判断还不够稳。

- 我是否定义了清楚的用户，而不是“所有人都能用”。
- 我是否定义了单一主任务，而不是一串功能列表。
- 我是否说明了每个工具为什么必要。
- 我是否说明了哪些能力现在故意不做。
- 我是否说明了系统最少需要哪些状态。
- 我是否说明了这里到底需不需要记忆。
- 我是否说明了什么情况下先澄清而不是直接回答。
- 我是否说明了输出什么才算“真的有帮助”。

### 合格信号

你的方案读起来应该有这种感觉:

- 目标聚焦
- 边界清楚
- 工具克制
- 输出可判断

### 危险信号

如果你的方案里出现这些情况，说明还在过度设计:

- 一个系统同时承担 5 种主任务
- 工具列表比目标描述还长
- 明明只是 FAQ 助手，却接入长期记忆和复杂工作流
- 没写失败点，却写了很多“未来能力”

## 第三部分: 风险与失败自测

一个 `Stage 1` 合格学习者，不只会写“主路径成功”，还应该能先看到哪里最容易失败。

请检查自己是否能做到:

- 至少说出 5 个失败点
- 能区分理解失败、工具失败、流程失败、交互失败
- 能指出至少 3 个应该暂停给人确认的地方
- 能说明“如果失败了，谁来接管，怎么接管”

### 合格信号

- 失败点具体，不抽象
- 改进思路是系统层的，而不是只说“换个模型”
- 能指出用户体验层的失败，而不只盯技术失败

### 危险信号

- 所有失败都归因给模型
- 完全没有写“问题超出范围怎么办”
- 没有设计任何人工接管点

## 第四部分: 表达与复述自测

进入 `Stage 2` 前，你应该不看稿也能把一个最小 Agent 讲清楚。

请检查自己是否能脱稿回答:

- 什么是 Agent
- 为什么不是所有需求都要做成 Agent
- 什么时候需要工具
- 什么时候需要状态
- 记忆和知识库为什么不同
- 为什么高风险动作要停下来给人确认

### 合格信号

- 回答里有系统视角
- 有你自己的例子
- 能解释“为什么”，而不只是“是什么”

### 危险信号

- 回答全是定义，没有场景
- 一离开课程原句就不会说了
- 只能讲功能，讲不出边界

## 阶段答辩题

下面这组问题可以作为 `Stage 1` 的正式放行答辩题。

### 必答题

1. 如果一个系统只需要回答问题，为什么它不一定需要 Agent。
2. 如果一个系统要调用搜索、数据库和发消息工具，它的风险点会出现在哪里。
3. 如果一个系统跨多轮任务工作，为什么需要状态。
4. 为什么长期记忆不应该等于“把所有对话都存起来”。

### 进阶题

1. 一个“看起来很聪明”的 Agent，为什么仍然可能不适合上线。
2. 为什么项目应该先设计人工介入点，而不是最后再补。
3. 为什么最小可行版本对 Agent 特别重要。

### 合格标准

- 每题都能说出场景化理由
- 每题都能说出边界或风险
- 不只是复述概念，而是能做判断

## 项目产物检查

进入下一阶段前，建议你至少完成下面这些产物:

- 1 份最小 Agent 设计画布
- 1 个引导项目拆解结果
- 1 组挑战题答案
- 1 次口头或书面答辩
- 1 份课后项目方案

如果这些材料都没有，说明你更像“读过 Stage 1”，但还不像“学会了 Stage 1”。

## 放行标准

只有当你已经具备下面这种意识时，`Stage 2` 才会真正有效:

> 先做结构判断，再做框架实现。

更具体一点说，你至少要能做到:

- 不把提示词当成系统设计本身
- 不为了显得高级而盲目上工具和多 Agent
- 不把状态、记忆、知识库混成一团
- 不忽略失败路径和人工接管点
- 能先做最小版本，再考虑扩展能力

## 最后一个问题

在进入 `Stage 2` 之前，先问自己一句:

> 我现在学到的是“怎么调用框架”，还是“怎么判断一个 Agent 系统应该怎么设计”。

如果你能明确回答后者，说明 `Stage 1` 的作用已经开始生效。
