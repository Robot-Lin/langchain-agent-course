# 05 Challenges

这一部分开始把学习者从“跟着理解”推进到“主动完成”。

如果说前面的概念和方法是在帮你建立判断框架，那么这里的练习就是在逼你把这些判断真正说出来、写出来、做出来。

## 这一章怎么用

建议你不要跳着做，而是按下面顺序推进:

1. 先做快速判断题
2. 再做需求拆解题
3. 再做复杂度压缩题
4. 再做失败分析题
5. 最后做口头答辩和课后项目

每做完一题，都建议你让 Codex 扮演“评审”，而不是“代做者”。

## Challenge 1: 判断题型练习

### 目标

训练你快速判断: 一个需求到底是不是 Agent 问题。

### 任务

请分别判断下面 4 个需求是否真的需要 Agent，并写出理由。

### 题目 A

用户上传一段文本，系统输出固定 JSON 结构。

### 题目 B

用户希望系统自动查找资料、整理答案、必要时追问，最后给出带来源的总结。

### 题目 C

用户点击按钮后，系统把表单内容发给后端保存。

### 题目 D

用户希望系统根据任务状态决定是否继续下一步、是否请求人工确认、是否调用外部系统。

### 你至少要回答

- 是否值得设计成 Agent
- 为什么
- 如果不是，更适合什么实现方式
- 如果是，最小需要什么能力

### 这题在练什么

- 场景识别
- 过度设计预防
- “模型可用”与“应该做 Agent”之间的区分

### 合格标准

- 不只回答“是”或“不是”
- 能说出判断背后的结构原因
- 能给出更合适的替代实现思路

### 推荐对 Codex 说

- 我会自己先判断这 4 题，请你只指出我哪道题的理由站不住。
- 不要直接公布答案，先从边界和任务类型角度挑战我。

## Challenge 2: 需求拆解练习

### 目标

训练你把一个模糊需求拆成最小 Agent 设计画布。

### 任务

从下面 3 个场景中任选 1 个，完成一份最小 Agent 设计画布:

- 学习教练
- 个人日程助理
- 研究资料整理助手

### 你至少要写清

- 用户
- 目标
- 输入
- 工具
- 状态
- 记忆
- 输出
- 风险
- 人工介入点

### 这题在练什么

- 从模糊想法到结构化设计
- 状态、记忆、工具的职责区分
- 在实现前完成系统定义

### 合格标准

- 目标是单一主任务，而不是功能堆叠
- 每个工具都有“为什么需要”的理由
- 状态与记忆没有混淆
- 至少写出 3 个风险点

### 推荐对 Codex 说

- 请像设计评审一样，检查我这张画布里最模糊的 3 个地方。
- 只提问题，不要帮我重写成标准答案。

## Challenge 3: 压缩复杂度练习

### 目标

训练你从“全能幻想”退回到“最小可行”。

### 任务

把下面这个想法压缩成最小可行版本:

> 做一个全能个人助手，能查资料、写邮件、管日程、读文件、记住偏好、自动执行工作流。

### 约束

你必须:

- 只保留一个主任务
- 只保留一个用户类型
- 只保留一个最必要工具
- 写出为什么删掉其他能力

### 这题在练什么

- 减法设计
- 最小可行 Agent 意识
- 抗拒“听起来很强但其实很散”的方案

### 合格标准

- 压缩后的目标依然清楚、有价值
- 删除的功能有明确理由
- 没有为了显得高级而保留多余能力

### 推荐对 Codex 说

- 帮我看看我是不是真的做了减法，还是只是换了种说法继续堆功能。
- 请从“单一主任务”角度挑战我的方案。

## Challenge 4: 失败复盘练习

### 目标

训练你把 Agent 失败看成系统问题，而不是只看模型答错。

### 任务

假设一个 Agent 出现了下面 3 种情况，请判断它们分别属于什么类型的失败，并写出至少一个改进思路。

- 回答很流畅，但引用了错误知识
- 自动调用了不该调用的工具
- 明明没有完成任务，却说已经完成

### 你至少要回答

- 这属于哪类失败
- 为什么会发生
- 系统层应该怎么改，而不是只说“换个模型试试”

### 这题在练什么

- 失败分类能力
- 系统改进意识
- 从结果回推设计缺口

### 合格标准

- 能区分理解失败、工具失败、流程失败、交互失败
- 改进思路针对系统结构，而不是停留在泛泛而谈

### 推荐对 Codex 说

- 请把我写的失败分析当成评审材料，挑出最空泛的地方。
- 帮我看我是不是把所有问题都粗暴归因给模型了。

## Challenge 5: 口头答辩练习

### 目标

训练你不用照稿，也能说清楚 Stage 1 的核心理解。

### 任务

请尝试脱稿回答下面这些问题:

- 什么是 Agent
- Agent 和聊天机器人的差别是什么
- 什么情况下必须引入工具
- 什么情况下必须引入状态
- 记忆和知识库为什么不是一回事
- 为什么高风险动作要设置人工确认

### 建议做法

- 先自己口头回答
- 再把答案写成简短提纲
- 最后让 Codex 只指出逻辑漏洞，不替你重写

### 合格标准

- 能说出自己的理解，而不是只复述术语
- 回答里有系统视角，而不是只谈 prompt
- 能举出至少一个自己的例子

## Challenge 6: 综合小作业

### 目标

把前面练习过的能力拼起来，完成一个最小设计说明。

### 任务

任选一个你熟悉的真实场景，写出一份 `200-400` 字的最小 Agent 方案说明，并附带:

- 最小工具清单
- 最小状态清单
- 至少 5 个失败点
- 至少 3 个人工介入点

### 这题在练什么

- 把零散概念收束成完整方案
- 在没有代码的前提下完成系统设计
- 为 Stage 1 课后项目做热身

### 合格标准

- 方案足够清楚，别人看完能知道你要做什么
- 结构紧凑，没有明显功能堆砌
- 能看出你已经具备最小系统设计意识

## 自评 Rubric

做完这一章后，你可以按下面 4 个维度给自己打分，每项 `1-5` 分:

- 判断力: 我能否判断一个需求是否值得做成 Agent
- 边界感: 我能否控制复杂度，不乱加工具和能力
- 结构感: 我能否分清输入、工具、状态、记忆、输出
- 风险感: 我能否提前看到失败路径和人工接管点

### 粗略参考

- `16-20` 分: 可以进入课后项目
- `12-15` 分: 建议再做一轮挑战题
- `11` 分及以下: 建议回看 `02-concepts.md` 和 `03-methods.md`

## 推荐学习闭环

每完成一个 challenge，都做一次小闭环:

1. 写下你的答案。
2. 让 Codex 只做评审。
3. 记录它指出的最关键问题。
4. 用你自己的语言修正答案。
5. 总结这题帮你建立了什么判断标准。
