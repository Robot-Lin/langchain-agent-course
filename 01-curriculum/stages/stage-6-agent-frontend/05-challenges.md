# 05 Challenges

这一章的 challenge 目标，是逼自己把“系统能力”翻译成“可用体验”。每道题都建议先自己做，再让 Codex 站在体验评审角度反馈。

## Challenge 1: 界面判断

请判断下面哪些需求只靠一个聊天框很难做好：

### 题目 A

一次性短问短答。

### 题目 B

长任务执行，中途需要用户审批。

### 题目 C

简单文本改写。

### 题目 D

需要展示多阶段计划、执行状态和结构化研究结果的系统。

## Challenge 2: 分区设计练习

请为一个 Agent 产品设计前端分区，至少包含：

- 输入区
- 计划区
- 状态区
- 审批区
- 结果区
- 历史区

要求：

- 每个区都写“为什么必须单独可见”
- 每个区都写“如果挤进聊天流会出什么问题”

## Challenge 3: 结果展示练习

请把一个“研究结果”设计成结构化前端展示。

要求：

- 至少拆成 4 个区块
- 说明每个区块为什么不能只放在聊天消息里

## Challenge 4: 审批交互练习

请为一个高风险动作设计审批 UI。

要求：

- 用户在审批前能看见什么
- 用户可以做哪些操作
- 拒绝后系统应该如何反馈
- 修改后系统应该如何继续

## Challenge 5: 体验复盘练习

从前面阶段选一个你做过的 Agent，回答：

- 如果给它做前端，最需要被看见的 3 类信息是什么
- 哪些地方最需要可控性交互
- 哪些地方最不应该继续藏在聊天泡泡里

## Challenge 6: Streaming 设计练习

请为一个多阶段任务型 Agent 设计 streaming 方案。

要求：

- 区分文本流、状态流、工具流和审批等待流
- 说明哪些内容适合实时更新，哪些适合阶段总结

## Challenge 7: History 与来源感设计

请为一个带 RAG 和 tools 的 Agent 设计历史和来源呈现方案。

要求：

- 用户能看见结果来自哪里
- 用户能区分历史上下文、检索结果和动作结果

## Challenge 8: 综合小项目

需求：

设计一个“企业 Agent 仪表盘”前端。它需要：

- 发起任务
- 看到计划
- 看到执行状态
- 在关键动作前审批
- 看到结构化结果与来源

交付要求：

- 1 张前端结构图
- 1 份任务旅程说明
- 1 份 streaming 方案
- 1 份审批与结果展示说明

## 自评 Rubric

完成本章 challenge 后，你至少要能做到：

- 不再把 Agent 前端理解成聊天框皮肤
- 能识别哪些信息必须单独可见
- 能为复杂任务设计清晰的状态与审批承接点
- 能把结果组织成结构化展示，而不是继续堆文本
