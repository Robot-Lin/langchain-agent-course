# 09 Case Comparisons

这一章最适合通过对照看清前端边界。下面这些案例，帮助你判断什么时候聊天框够用，什么时候必须升级为真正的 Agent 产品界面。

## 案例一: 单轮问答助手

需求：

- 用户提问。
- 系统回答。

重点判断：

- 增强版 chat UI 通常够用。
- 状态和审批需求很弱。
- 结果展示可以简单。

## 案例二: 深度研究 Agent

需求：

- 用户下达复杂研究任务。
- 系统分阶段执行。
- 中间产生计划、子任务和多轮结果。

重点判断：

- 单纯聊天流很快就会失控。
- 需要计划区、状态区、结果区和历史区。
- streaming 需要承担状态反馈，而不是只冒字。

## 案例三: 企业审批 Agent

需求：

- 系统读取信息。
- 系统建议动作。
- 用户决定是否批准执行。

重点判断：

- 审批入口必须明确可见。
- 用户必须在审批前看到动作预览与风险说明。
- 拒绝或修改后的继续路径也必须清楚。

## 案例四: 带 RAG 和 tools 的支持 Agent

需求：

- 系统检索知识。
- 系统查询业务状态。
- 系统可能执行动作。

重点判断：

- 前端需要明确来源感。
- 用户要能区分“文档依据”“系统查询”“动作结果”。
- 结果展示应当结构化，而不是混在一段话里。

## 案例五: 为什么很多 Agent 产品看起来不好用

常见原因：

- 只做了一个聊天框。
- 没有状态反馈。
- 没有审批承接。
- 没有把结果结构化。
- 没有让用户知道系统依据和来源。

## 本章对照练习

拿你自己的一个 Agent 想法，分别回答：

- 用户最需要看见的状态是什么。
- 用户最需要控制的动作是什么。
- 用户最需要理解的结果来源是什么。
- 哪些内容不应该继续塞进聊天流。

如果这四个问题你都能答清，Stage 6 就真正开始落地了。
