# 07 Codex Workshop

这一章里，Codex 最适合扮演的不是“帮你画一个酷炫页面”，而是体验结构评审员、可控性交互教练和前端信息架构纠偏器。

## Codex 在 Stage 6 里最适合的 5 个角色

### 1. Agent UX 解释器

适合帮你区分：

- 聊天 UI 和 Agent UI
- 消息流和状态流
- 普通结果展示和结构化结果展示
- 技术 interrupt 和用户审批体验

推荐提问：

```text
请你用一个真实 Agent 产品例子，帮我解释为什么 Agent UX 不等于聊天 UX。不要只列点，要一步步说明用户在哪些时刻会失去控制感。
```

### 2. 信息架构评审员

适合在你已经有初版前端结构后，让 Codex 找出分区缺失和信息挤压问题。

推荐提问：

```text
这是我给这个 Agent 设计的前端分区。请你按 Stage 6 的思路，先检查是否缺关键区域，再检查哪些信息仍然被错误塞进聊天流里，最后告诉我最优先该补哪一块。
```

### 3. Streaming 与状态设计教练

适合让 Codex 帮你判断 streaming 是否真的承担了反馈角色。

推荐提问：

```text
请你站在 Agent UX 的角度，检查我这套 streaming 方案是不是只在做打字效果，还是已经真正承担了状态反馈和进度解释的作用。
```

### 4. 审批与可控性交互评审员

适合让 Codex 站在严格产品体验角度，检查你的审批设计是否清楚。

推荐提问：

```text
请你站在严格产品评审角度，检查这个审批界面。优先指出用户在哪些地方会不知道该批什么、为什么批、批完后会发生什么。
```

### 5. 课后项目导师

课后项目阶段，不要让 Codex 直接给完整页面方案，而要让它只做引导和评审。

推荐提问：

```text
我正在做 Stage 6 的课后项目。请不要直接给我完整界面答案，只根据验收标准检查我的分区、状态、审批、结果展示和历史设计，并一步步告诉我下一轮该优先改什么。
```

## 推荐学习顺序

1. 先自己画任务旅程。
2. 再让 Codex 检查体验断点。
3. 先自己做前端分区。
4. 再让 Codex 检查缺层和信息混挤。
5. 先自己设计 streaming、审批和结果展示。
6. 最后让 Codex 做正式体验评审。

## 常见错误用法

### 错误一

一上来就让 Codex “帮我设计完整前端页面”。

问题：

- 你会直接跳过最重要的任务旅程和信息结构训练。

### 错误二

只让 Codex 解释概念，不让它评审你的页面结构。

问题：

- 你会以为自己懂了，但无法迁移到真正的产品设计。

### 错误三

把 challenge 题直接交给 Codex 完成。

问题：

- 你失去的恰恰是 Stage 6 最核心的体验判断训练。

## Stage 6 推荐协作闭环

- 你先写任务旅程。
- 让 Codex 做体验断点评审。
- 你再画前端分区。
- 让 Codex 做信息架构评审。
- 你再补 streaming、审批和结构化展示。
- 让 Codex 做最终放行评审。

## 一次完整对练模板

```text
我正在学习 Stage 6: Agent Frontend。
这是我的 Agent 产品需求和当前前端结构草案。
请你不要直接替我做完整答案。
先按以下顺序帮助我：
1. 检查我的任务旅程里哪些时刻用户会失去方向感。
2. 检查我缺失的前端分区。
3. 检查我的 streaming 是否真的反馈了系统状态。
4. 检查我的审批和结果展示哪里不够清楚。
5. 最后只告诉我下一轮该优先修改什么。
```
