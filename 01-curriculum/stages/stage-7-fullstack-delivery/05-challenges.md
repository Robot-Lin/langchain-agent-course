# 05 Challenges

这一章的 challenge 目标，是逼自己从“方案思维”进入“交付思维”。每道题都建议先自己做，再让 Codex 站在交付评审角度反馈。

## Challenge 1: 链路判断

请为一个全栈 Agent 产品写出从用户点击到最终结果的完整链路。

要求至少包含：

- 前端
- 请求通路
- Agent runtime
- 数据或知识层
- 流式反馈
- 评测与追踪

## Challenge 2: 原型还是交付

请判断下面哪些项目仍然只是原型，哪些已经开始具备交付条件。

### 题目 A

只有本地脚本，没有前端，没有追踪。

### 题目 B

有前端、有流式链路、有 Agent runtime、有追踪和验收指标。

### 题目 C

有很多功能，但没有办法稳定复现和排障。

## Challenge 3: 选型练习

请为一个全栈 Agent 产品写一段选型说明。

要求说明：

- 为什么主要用 LangChain / LangGraph / Deep Agents 中的哪一层
- 为什么当前前端形态合理
- 为什么当前部署方式合理

## Challenge 4: 评测练习

请为一个交付中的 Agent 产品写出：

- 3 个交付验收指标
- 3 个运行追踪点
- 3 个上线后应持续关注的问题

## Challenge 5: 全栈复盘练习

从前面阶段选一个你做过的 Agent，回答：

- 如果把它真正交付出来，还缺哪几层
- 最大的交付风险在哪里
- 最该先补哪一层，为什么

## Challenge 6: Agent Server 边界练习

请说明：

- 哪些能力适合通过 Agent Server 暴露
- 哪些能力仍然应该保留在普通业务 API 中
- 为什么这两者不能完全混为一谈

## Challenge 7: 交付验收练习

请为一个“研究控制台产品版”写一份最小验收标准。

要求覆盖：

- 链路是否完整
- streaming 是否可用
- 状态是否可见
- 结果是否可理解
- 评测是否可执行

## Challenge 8: 综合小项目

需求：

设计一个“全栈 Agent 产品交付方案”。它需要：

- 有前端体验
- 有 Agent runtime
- 有知识与数据层
- 有流式反馈
- 有 deployment、observability 和 evaluation 思路

交付要求：

- 1 张全栈系统图
- 1 份交付链路说明
- 1 份验收与评测说明
- 1 份选型说明

## 自评 Rubric

完成本章 challenge 后，你至少要能做到：

- 不再把交付理解成“把功能做完”
- 能把系统按完整链路解释出来
- 能提出交付级指标和验收标准
- 能说明为什么当前方案适合当前产品阶段
