# 07 Codex Workshop

这一章里，Codex 最适合扮演的不是“帮你写一份漂亮架构文档”，而是交付评审员、全链路纠偏器和选型陪练。

## Codex 在 Stage 7 里最适合的 5 个角色

### 1. 全链路解释器

适合帮你区分：

- 前端、后端、Agent runtime、数据层之间的关系
- Agent Server 和普通 API 的边界
- 原型链路和交付链路的差别

推荐提问：

```text
请你用一个真实 Agent 产品例子，帮我解释从用户发起请求到系统返回结果的完整交付链路。不要只列模块，要一步步说明哪些环节让它从 demo 变成产品。
```

### 2. 系统组装评审员

适合在你已经有初版系统图后，让 Codex 找缺层和断链问题。

推荐提问：

```text
这是我给这个 Agent 产品画的全栈链路。请你按 Stage 7 的思路，先检查是否缺关键层，再检查哪些连接还没有闭环，最后告诉我最优先该补哪一段。
```

### 3. 交付验收教练

适合让 Codex 帮你判断验收标准是不是空洞。

推荐提问：

```text
请你站在产品交付评审角度，检查我这套验收指标是否真的能说明系统可以交付。优先指出哪些指标无法反映真实产品质量。
```

### 4. 选型评审员

适合让 Codex 检查你的框架和部署选择有没有明确取舍逻辑。

推荐提问：

```text
请你站在严格架构评审角度，检查我这份选型说明。优先指出哪些地方只是惯性选择，没有解释为什么当前产品阶段适合这样做。
```

### 5. 课后项目导师

课后项目阶段，不要让 Codex 直接给完整交付方案，而要让它只做引导和评审。

推荐提问：

```text
我正在做 Stage 7 的课后项目。请不要直接给我完整方案，只根据验收标准检查我的全链路设计、deployment、observability、evaluation 和选型说明，并一步步告诉我下一轮该优先补什么。
```

## 推荐学习顺序

1. 先自己画完整链路。
2. 再让 Codex 检查断链点。
3. 先自己写交付与运行方案。
4. 再让 Codex 检查是否可追踪、可评测、可迭代。
5. 先自己写选型说明。
6. 最后让 Codex 做正式交付评审。

## 常见错误用法

### 错误一

一上来就让 Codex “帮我设计完整交付架构”。

问题：

- 你会直接跳过最重要的交付判断训练。

### 错误二

只让 Codex 解释概念，不让它评审你的系统链路。

问题：

- 你会以为自己懂了，但无法迁移到真实交付。

### 错误三

把 challenge 题直接交给 Codex 完成。

问题：

- 你失去的恰恰是 Stage 7 最核心的交付判断训练。

## Stage 7 推荐协作闭环

- 你先画完整链路。
- 让 Codex 做断链评审。
- 你再补 deployment、observability 和 evaluation。
- 让 Codex 做交付可行性评审。
- 你再补选型说明和验收标准。
- 让 Codex 做最终放行评审。

## 一次完整对练模板

```text
我正在学习 Stage 7: Full-stack Delivery。
这是我的 Agent 产品需求和当前交付链路草案。
请你不要直接替我做完整答案。
先按以下顺序帮助我：
1. 检查我的全链路里哪里还没有闭环。
2. 检查我缺失的 deployment、observability 和 evaluation 部分。
3. 检查我的验收标准是不是能真正反映可交付性。
4. 检查我的选型说明哪里不够有说服力。
5. 最后只告诉我下一轮该优先修改什么。
```
