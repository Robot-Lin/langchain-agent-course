# 04 Guided Project

`Stage 4` 的引导项目选择 `深度研究 Agent`，因为它最能体现 Deep Agents 的真正价值: 难点不只是流程怎么走，而是目标怎么拆、上下文怎么分、角色怎么协作、结果怎么收束。

这一章的重点不是把它做成“自动研究公司”，而是让学习者第一次真正把复杂目标整理成一个可执行、可治理的复杂任务型 Agent 结构。

## 项目定位

### 你要设计的是什么

你要设计一个围绕开放研究主题运行的复杂任务型 Agent。系统需要:

- 接收开放研究主题
- 规划研究路线
- 拆出多个子任务角色
- 分层组织上下文
- 在关键阶段汇总结果
- 在关键节点允许人工修正方向
- 最终生成围绕总目标收束的研究结论

### 你现在不追求什么

这一阶段故意不追求:

- 多到失控的子代理数量
- 过度复杂的工程化实现
- 完整前端产品化
- 夸张的“完全自动化团队”幻想

因为 `Stage 4` 真正要训练的是: 如何把一个复杂目标组织成一个可执行、可管理、可治理的 Agent 结构。

## 这个项目真正训练什么

这个项目表面上是研究 Agent，实际上训练的是 6 种能力:

- 复杂目标拆解
- planning 设计
- context engineering
- subagent 必要性判断
- 复杂任务治理
- 为什么这里更适合 Deep Agents 的边界判断

## 最终交付物

完成这个引导项目后，你应该至少交出以下内容:

- 一份复杂目标拆解图
- 一份阶段目标说明
- 一份 context 分层说明
- 一份 subagent 分工表
- 一份汇总与回收机制说明
- 一份治理点与审核点清单
- 一段“为什么这里更适合 Deep Agents”的边界说明

## 项目步骤

## Step 1: 定义复杂总目标

### 这一步要解决什么

先把研究目标定义清楚，否则后面所有角色和上下文都会散。

### 你要回答的问题

- 这个研究系统最终要交付什么
- 成功的研究结果长什么样
- 这个目标为什么不是简单问答或单次检索

### 常见错误

- 目标写成“帮我做研究”这种模糊大词
- 目标里同时混入报告生成、产品建议、执行落地等多个维度
- 还没明确最终交付物，就先想角色

### 这一小步的产出

- 一句复杂总目标说明
- 一段最终交付结果说明

## Step 2: 拆阶段目标

### 这一步要解决什么

你要把总目标拆成更可管理的阶段，而不是让系统抱着一个巨大目标一路乱跑。

### 你要回答的问题

- 这个复杂任务天然会分成哪些阶段
- 各阶段的主要产出是什么
- 哪些阶段之间有明确依赖关系

### 常见错误

- 阶段太多，反而失去结构
- 阶段只是动作列表，没有产出定义
- 阶段边界不清，导致后续角色设计混乱

### 这一小步的产出

- 一份阶段目标列表

## Step 3: 设计 context 分层

### 这一步要解决什么

你要决定不同类型上下文该如何被组织，而不是全部堆进同一个 prompt。

### 你要回答的问题

- 哪些信息属于长期背景资料
- 哪些属于当前任务状态
- 哪些只属于某个子任务角色
- 哪些内容必须被汇总上收，而不是横向扩散

### 常见错误

- 所有角色共享全部资料
- 长期资料、当前任务状态和子任务指令混成一团
- 汇总层拿到的是一堆零散材料而不是结构化中间结果

### 这一小步的产出

- 一份 context 分层说明

## Step 4: 设计 subagents

### 这一步要解决什么

你要判断哪些子任务真的需要独立角色，而不是为了“像团队”而拆角色。

### 你要回答的问题

- 哪些子任务天然独立
- 哪些角色需要独立上下文
- 哪些角色有不同视角或责任
- 主代理负责什么，子代理负责什么

### 常见错误

- 角色越拆越多，但没有真正降低复杂度
- 主代理仍然持有全部细节，子代理只是“换个名字”
- 角色边界不清，产出重复或冲突

### 这一小步的产出

- 一份 subagent 分工表

## Step 5: 设计汇总与回收机制

### 这一步要解决什么

复杂任务不是“拆完了就结束”，更难的是把子任务结果收回来，围绕总目标重新组织。

### 你要回答的问题

- 子代理分别返回什么
- 哪些结果直接进入下阶段
- 哪些结果必须被主代理汇总
- 最终输出如何围绕总目标收束

### 常见错误

- 所有子任务都产出很多内容，但没人负责收束
- 汇总阶段只是把内容拼起来
- 最终结果越来越长，却越来越不聚焦

### 这一小步的产出

- 一份汇总与回收机制说明

## Step 6: 设计治理点

### 这一步要解决什么

你要防止复杂任务慢慢偏航。

### 你要回答的问题

- 哪些阶段必须检查是否跑偏
- 哪些中间结果必须人工审核
- 哪些子任务失败后应被回收或重做
- 最终输出前需要做什么质量检查

### 常见错误

- 只管拆任务，不管收敛
- 把治理当成最后补丁
- 认为复杂任务只要能跑起来就行

### 这一小步的产出

- 一份治理点与审核点清单

## Step 7: 写出边界说明

### 这一步要解决什么

你要明确说明: 为什么这里更适合 Deep Agents，而不是普通 LangGraph 工作流。

### 你要回答的问题

- 这个任务的难点主要落在哪
- 为什么仅有流程控制不够
- 为什么需要 planning、context 分层和 subagents

### 这一小步的产出

- 一段“为什么这里更适合 Deep Agents”的说明

## Step 8: 用 Codex 做复杂任务评审

### 推荐你怎么配合 Codex

建议这样使用 Codex:

- 先帮我检查我的总目标是不是还不够清楚，不要直接重写。
- 只从 context 分层角度审查我的设计，不要帮我加更多角色。
- 只从 subagent 必要性角度批评我的方案。
- 帮我看看我的汇总机制是不是会让结果越来越散。

### 一个很好的带学节奏

1. 你先回答一个步骤。
2. 让 Codex 指出 2 到 3 个关键问题。
3. 你修正后再继续。

## 阶段检查点

完成项目后，你至少应该能稳定说清楚:

- 这个复杂目标为什么需要拆层
- context 为什么不能混成一个大 prompt
- 哪些 subagents 是真必要的
- 为什么治理点对复杂任务型 Agent 很关键
- 为什么这个任务更适合 Deep Agents 而不是普通 LangGraph 工作流
