# 07 Codex Workshop

这个文件不是“给 Codex 的说明”，而是“给学习者的 Stage 4 协作手册”。它的目的不是让 Codex 替你把复杂任务方案直接做完，而是让你在和 Codex 的配合里，真正学会复杂任务组织。

## Stage 4 与 Codex 配合的核心原则

在这一阶段，Codex 最适合扮演这些角色:

- 复杂任务解释者
- 目标拆解教练
- context 分层审查员
- subagent 必要性审查员
- 汇总与治理评审员

在这一阶段，Codex 最不适合扮演的角色是:

- 直接替你完成整套复杂任务方案的人
- 用更多角色掩盖结构问题的人
- 在你还没说明总目标前就开始拆很多子代理的人

一句话原则:

> 在 Stage 4，Codex 应该帮你形成复杂任务组织判断，而不是替你跳过复杂任务设计。

## 使用前先告诉 Codex 什么

每次开始一个会话前，建议你至少告诉 Codex 这 5 件事:

1. 我现在处于 `Stage 4`。
2. 我的目标是学习复杂任务组织，不是直接拿完整方案。
3. 请先问我问题，再帮我评审。
4. 不要一开始就替我拆很多角色。
5. 如果你觉得这个任务其实还不需要 Deep Agents，请明确提醒我。

一个通用开场模板:

```text
我现在在学习 LangChain Agent Course 的 Stage 4。
这一阶段我想训练的是复杂任务型 Agent 设计，不是直接拿完整答案。
请你先做导师和评审，不要直接给我完整方案。
每次只推进一步，先问我问题，再指出我的结构问题。
如果你觉得我这个任务其实还没到必须用 Deep Agents 的程度，请明确告诉我原因。
```

## 模式一: 复杂任务概念讲解模式

适用场景:

- 你刚读完概念，但还不真正理解 Deep Agents 解决什么问题
- 你知道 planning、context engineering、subagents 这些词，但讲不出自己的话
- 你还说不清 Deep Agents 和 LangGraph 的边界

推荐提问:

```text
请用课程学习者能听懂的方式解释为什么这里需要 Deep Agents，而不是继续用普通 LangGraph 工作流。
不要只讲定义，要结合真实任务说明。
```

```text
请用一个复杂研究任务案例解释 planning、context engineering 和 subagents 之间的关系。
重点讲系统为什么需要这样组织，不要只讲术语。
```

```text
请解释为什么 context engineering 不是“塞更多上下文”。
请用一个反例帮助我理解。
```

## 模式二: 复杂目标拆解模式

适用场景:

- 你已经有一个复杂目标，但不知道怎么拆成阶段和角色
- 你不确定主代理、子代理和汇总目标怎么设计
- 你想练复杂目标拆解法

推荐提问:

```text
我想把这个复杂目标设计成 Deep Agents 系统。
请先不要直接给完整方案，先按总目标、阶段目标、子任务角色、汇总目标这几个维度一步一步带我拆解。
每一步先提问，不要直接替我填答案。
```

```text
请先判断这个任务现在是否真的适合 Deep Agents。
如果不适合，请明确告诉我最主要的原因。
```

## 模式三: context 分层评审模式

适用场景:

- 你总觉得上下文越写越多
- 你不确定哪些信息该给所有角色，哪些只该给局部任务
- 你怀疑自己还是在把所有资料塞进一个大 prompt

推荐提问:

```text
请只从 context 分层角度审查我的设计。
告诉我哪些信息该共享，哪些信息该隔离，哪些信息其实不该一直存在上下文里。
不要帮我加更多角色。
```

```text
请帮我判断我的上下文分层是否真的减少了复杂度，还是只是换了个名字继续堆信息。
```

## 模式四: subagent 必要性评审模式

适用场景:

- 你不确定角色是不是拆多了
- 你担心自己只是为了“看起来像团队”而拆角色
- 你想知道哪些角色真正降低了主代理复杂度

推荐提问:

```text
请只从 subagent 必要性角度审查我的方案。
告诉我哪个角色最值得删，哪个角色是真正必要的。
不要帮我增加新的角色。
```

```text
请判断这些子代理是否真的有独立职责、独立上下文和独立产出。
```

## 模式五: 汇总与治理评审模式

适用场景:

- 你不确定结果会不会越来越散
- 你担心系统拆得很细，但最后收不回来
- 你想知道治理点放得够不够

推荐提问:

```text
请只从汇总与治理角度审查我的方案。
告诉我哪里最容易偏航，哪里最应该加审核点，哪里最容易出现“子任务都做了但整体散掉”的情况。
```

```text
请判断我的回收机制是不是会让结果越来越长但越来越不聚焦。
```

## 模式六: 项目带学模式

适用场景:

- 你在做 [04-guided-project.md](./04-guided-project.md)
- 你在做 [05-challenges.md](./05-challenges.md)
- 你准备进入课后项目

推荐提问:

```text
请按 Stage 4 的知识点顺序带我完成深度研究 Agent。
每次只推进一步，并先告诉我这一步对应哪个概念和方法。
不要一次给完整方案。
```

```text
我现在做的是 Stage 4 课后项目。
请不要代做，只按目标拆解、context、subagents、汇总与治理做导师和评审。
```

## 模式七: 模拟答辩模式

适用场景:

- 你准备做 Stage 4 放行检查
- 你想测试自己是否已经能清楚讲出复杂任务组织思路
- 你希望进入 Stage 5 前做一次复杂任务压力测试

推荐提问:

```text
你来当导师，逐题提问我 Stage 4 的答辩题。
不要先给答案，等我回答完再指出哪里不清楚。
```

```text
请重点追问我，为什么这个任务更适合 Deep Agents，以及我的治理点如何防止任务偏航。
```

## 推荐学习动作

一个很稳的对练闭环是:

1. 先让 Codex 解释。
2. 再自己复述。
3. 再让 Codex 挑错。
4. 再修正你的表达。
5. 最后让 Codex 模拟复杂任务评审或答辩。

这个顺序会逼你从“看懂复杂任务结构”进入“能解释为什么这样组织复杂任务”。

## 什么时候不要让 Codex 直接回答

在下面这些场景里，最好不要让 Codex 直接给完整答案:

- 你正在做 challenge
- 你正在做课后项目
- 你准备做答辩
- 你还没有自己写过第一版复杂任务方案

更好的做法是:

- 先写你的版本
- 再让 Codex 做评审
- 最后自己再改一版

## 一个完整的 Stage 4 会话模板

你可以直接复制下面这段开始一轮高质量学习:

```text
我现在在学习 LangChain Agent Course 的 Stage 4。
我希望你扮演导师和评审，而不是直接代做。
请先根据我给你的任务，判断它是否真的已经适合 Deep Agents。
如果适合，再按总目标、阶段目标、context 分层、subagents、汇总与治理这几个维度，一步一步带我拆解。
每次只推进一步。
先问我问题，再指出我回答里的结构问题。
不要直接替我写完整方案，也不要一次给完整答案。
如果你觉得这个任务其实更适合留在 LangGraph，请明确告诉我原因。
```

## Stage 4 结束前，你应该达到的协作状态

如果这一章学得比较扎实，到了最后你和 Codex 的关系应该是:

- 你负责提出复杂任务草案
- Codex 负责挑战你的目标拆解、上下文和角色设计
- 你负责修正汇总与治理机制
- Codex 负责指出哪些复杂度是真必要的
- 你最终能不依赖 Codex，也把一个复杂任务型 Agent 讲清楚

这才说明 Codex 真正起到了“放大学习效率”的作用，而不是替你绕过了复杂任务思考过程。
