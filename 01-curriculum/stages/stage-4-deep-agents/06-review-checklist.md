# 06 Review Checklist

这个文件的作用不是让你“打卡完成 Stage 4”，而是帮你判断: 你到底有没有真正掌握复杂任务型 Agent 的组织思维，以及你是否已经准备好进入工程化整合阶段。

## 这一章怎么用

建议你分 3 次使用这个清单:

1. 学完概念和方法后，做第一次自测。
2. 做完引导项目后，检查复杂任务结构是否清楚。
3. 做完挑战题和课后项目后，做正式放行判断。

真正有效的用法不是机械打勾，而是要求自己: 每一项都能举出一个自己的例子，或者一个明确反例。

## 第一部分: 复杂任务概念理解自测

如果下面任意 3 项做不到，建议不要急着进入 `Stage 5`。

- 我能解释 Deep Agents 解决的是什么复杂问题。
- 我能解释 Deep Agents 和 LangGraph 的关系与边界。
- 我能说明 planning 在复杂任务里的真正作用。
- 我能解释 context engineering 为什么不是简单塞更多上下文。
- 我能说明 subagents 为什么不是越多越好。
- 我能解释 complex task governance 为什么重要。
- 我能说出复杂任务失控的常见方式。
- 我能指出什么时候任务真的适合 Deep Agents。

### 合格信号

- 你不是只会说“Deep Agents 更强”
- 你能解释它强在哪里
- 你能说明复杂度主要落在哪一层

### 危险信号

如果你还有这些表现，建议回看概念部分:

- 把 Deep Agents 理解成“更多 Agent”
- 把 context engineering 理解成“喂更多资料”
- 把 subagents 理解成“角色越多越高级”

## 第二部分: 复杂任务结构自测

请对你自己的一个方案做检查。如果下面这些问题答不清，说明你还没有真正掌握复杂任务设计。

- 我是否定义了清楚的总目标。
- 我是否把总目标拆成了清楚的阶段目标。
- 我是否设计了 context 分层，而不是混成一个大 prompt。
- 我是否说明了哪些角色真的需要存在。
- 我是否说明了各角色输入什么、输出什么。
- 我是否设计了结果汇总和回收机制。
- 我是否说明了治理点放在哪里。
- 我是否说明了为什么这里更适合 Deep Agents。

### 合格信号

你的方案读起来应该有这种感觉:

- 总目标清楚
- 阶段有层次
- 上下文有组织
- 角色有边界
- 结果能收束
- 治理有抓手

### 危险信号

如果你的方案里出现这些情况，说明复杂任务设计还不稳:

- 目标很大，但没有拆层
- 所有角色都拿全部资料
- 角色很多，却没有职责差异
- 各子任务都在输出内容，但没人负责收束

## 第三部分: 风险与治理自测

一个 `Stage 4` 合格学习者，不只会拆任务，还应该能先看到复杂任务型 Agent 最容易失控在哪。

请检查自己是否能做到:

- 至少说出 5 个失控风险点
- 能区分目标拆解问题、context 问题、subagent 问题、治理问题
- 能指出至少 3 个关键治理点或审核点
- 能说明为什么这些风险不能只靠 prompt 修补

### 合格信号

- 风险点具体，不抽象
- 改进思路针对任务组织结构
- 能看到“慢慢跑偏”这种系统风险，而不只盯单次错误

### 危险信号

- 所有问题都归因给模型
- 完全没有写治理点
- 没有说明汇总与回收机制如何防散

## 第四部分: 表达与复述自测

进入 `Stage 5` 前，你应该不看稿也能把一个复杂任务型 Agent 讲清楚。

请检查自己是否能脱稿回答:

- 为什么这个任务更适合 Deep Agents
- planning、context engineering、subagents 分别解决什么问题
- 为什么上下文不能混成一个大 prompt
- 为什么角色越多不一定越好
- 为什么复杂任务治理很关键

### 合格信号

- 回答里有任务组织思维
- 有你自己的复杂任务例子
- 能解释“为什么这些角色要存在”“为什么这些上下文要分层”

### 危险信号

- 只能讲术语，讲不出任务结构
- 一离开教材原句就不会说了
- 只能说“更复杂了”，讲不出复杂在哪

## 阶段答辩题

下面这组问题可以作为 `Stage 4` 的正式放行答辩题。

### 必答题

1. 为什么这个复杂任务需要上下文分层。
2. 为什么这里的 subagents 是真必要，而不是炫技。
3. 为什么复杂任务的失控风险不只是“答错了”。
4. 为什么这个任务更适合 Deep Agents，而不是普通 LangGraph 工作流。

### 进阶题

1. 为什么 planning 的难点不只是列步骤。
2. 为什么一个“看起来很多角色”的系统仍然可能非常差。
3. 如果你把 Stage 3 的多阶段研究流程升级到 Stage 4，你会先补什么。

### 合格标准

- 每题都能结合复杂任务场景回答
- 每题都能提到组织边界或治理风险
- 不只是会复述定义，而是真的会判断

## 项目产物检查

进入下一阶段前，建议你至少完成下面这些产物:

- 1 份复杂目标拆解图
- 1 份 context 分层说明
- 1 份 subagent 分工说明
- 1 份汇总与回收机制说明
- 1 份治理点与审核点清单
- 1 组挑战题答案
- 1 次基于标准的自我答辩或 Codex 评审
- 1 份课后项目方案

如果这些产物都没有，说明你更像“读过 Stage 4”，而不是“掌握了 Stage 4”。

## 放行标准

只有当你已经能意识到下面这些事时，`Stage 5` 才会真正有效:

- 复杂任务的难点不仅是流程，而是任务组织本身
- context engineering 的核心是组织，而不是增量
- subagents 的价值是降复杂度，而不是造热闹
- 复杂任务必须有治理点和收束机制
- 有些任务必须先组织清楚，后面才谈工程化整合

更具体一点说，你至少要能做到:

- 不把复杂目标直接丢给一个主代理
- 不把所有上下文混成一个大 prompt
- 不让子代理数量失控
- 不忽略汇总与治理机制
- 能说清项目为什么更适合 Deep Agents

## 最后一个问题

在进入 `Stage 5` 之前，先问自己一句:

> 我现在学到的是“怎么把任务拆得更复杂”，还是“怎么让一个复杂目标被组织得更清楚、更可管理、更不容易失控”。

如果你能明确回答后者，说明 `Stage 4` 的关键价值已经开始内化。
