# 01 Overview

## 本章目标

Stage 5 的目标，是让学习者从“会做复杂 Agent”升级到“会设计可进入真实业务环境的 Agent 工程系统”。

完成这一章后，学习者应该能回答五个关键问题：

- 这个 Agent 的外部能力从哪里来。
- 这个 Agent 的知识从哪里来。
- 这个 Agent 的状态和数据落在哪里。
- 这个 Agent 怎么被追踪、评估和验证。
- 这个 Agent 怎么避免在真实业务里失控。

## 为什么 Stage 5 紧接 Stage 4

Stage 4 解决的是复杂任务组织：规划、上下文工程、子代理、长任务治理。
但如果没有工程化层，这些能力通常只能停留在：

- 单次演示
- 局部实验
- 很强的概念设计
- 很脆弱的落地能力

所以 Stage 5 要补上的，是让 Agent 真正接上世界的那几层：

- 工具和协议层
- 知识和检索层
- 数据和存储层
- 服务和接口层
- 追踪和评测层
- 权限和治理层

## 这一章和官方新版文档的关系

官方新版架构把很多能力拆得更清楚：

- LangChain 继续负责高层 agent 搭建与工具整合。
- MCP 负责标准化外部能力接入。
- memory 被明确拆成短期与长期。
- LangSmith 把 observability 和 evaluation 独立成工程闭环。

这意味着我们在这一章里学的，不是若干“配套技术”，而是 Agent 真实落地时必然出现的结构层。

## 学习节奏建议

### Session 1

- 建立工程化心智模型。
- 先把 MCP、RAG、memory、数据库、API 的边界彻底分开。

### Session 2

- 学分层设计方法。
- 学会从一个需求反推系统需要哪些层。

### Session 3

- 做引导项目。
- 重点练知识层、工具层、数据层和治理层。

### Session 4

- 用 challenge 做迁移训练。
- 让 Codex 站在架构评审和风险评审的角度做反馈。

## 本章产出物

学完 Stage 5，至少应该产出：

- 一张工程化 Agent 架构图。
- 一份知识层与工具层说明。
- 一份 memory / database / API 设计边界说明。
- 一份 LangSmith observability 与 evaluation 方案。
- 一份最小权限与审计清单。

## 进入 Stage 6 前的判断标准

如果学习者还不能清楚回答下面三件事，就不建议立刻进入前端体验层：

- 系统是如何连接知识、工具和业务数据的。
- 系统如何被追踪、观察和评分。
- 系统如何在有风险的动作前停下来并接受控制。
