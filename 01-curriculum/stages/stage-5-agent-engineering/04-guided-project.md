# 04 Guided Project

本章引导项目选择 `企业知识与流程一体化 Agent 后端架构`。它不是让学习者直接写很多代码，而是让学习者从产品需求出发，设计一个真正能接知识、接业务、接观测、接治理的 Agent 系统。

## 项目目标

设计一个企业内部 Agent 系统。它能够：

- 接收员工提问或流程请求。
- 从企业知识库中检索相关制度与文档。
- 在合适时通过 MCP 或受控工具调用外部系统。
- 通过 API 获取业务状态或写入业务动作。
- 对关键步骤进行追踪、评测和治理。

## 为什么这个项目适合 Stage 5

因为它同时迫使学习者处理五类边界：

- 文档知识边界
- 业务数据边界
- 工具能力边界
- 服务接口边界
- 风险动作边界

这正是 Stage 5 的核心训练目标。

## 项目最终交付物

完成这个项目时，至少要交付：

- 一张自然语言工程架构图
- 一份知识层设计说明
- 一份工具 / MCP / API 接入说明
- 一份数据与 memory 边界说明
- 一份 observability 与 evaluation 方案
- 一份最小权限与治理清单

## Step 1: 先把需求拆成系统层

不要先想接什么技术，先把需求拆成几个问题：

- 哪些问题要靠知识检索解决。
- 哪些问题要靠业务查询解决。
- 哪些问题要靠真实动作执行解决。
- 哪些环节需要人工确认。
- 哪些环节必须留下记录。

这一阶段只做结构判断，不做实现细节。

## Step 2: 设计知识层

明确：

- 哪些资料进入知识库。
- 检索结果需要怎样被引用。
- 哪些问题只允许根据受控资料回答。
- 知识更新后，系统应该如何保持最新。

这一部分对应 Stage 5 对 RAG 的理解。

## Step 3: 设计 memory、数据库和业务数据边界

在这个步骤里，要把三类东西分开：

- 用户偏好、长期设定、跨会话连续信息
- 可检索的文档知识
- 业务系统的真实状态和记录

如果这里分不清，后面整个系统都会变得难以治理。

## Step 4: 设计工具 / MCP / API 接入

先判断：

- 哪些能力适合暴露为 MCP。
- 哪些能力更适合作为内部 API。
- 哪些能力只读，哪些能力可写。
- 哪些动作必须被包装成受控接口，而不能直接暴露底层系统。

这一部分训练的是“接入能力”和“限制能力”同时存在的意识。

## Step 5: 设计 observability

至少要明确：

- 哪些运行步骤必须被 trace。
- 哪些失败必须被记录。
- 哪些关键中间结果要能回放。
- 哪些问题发生时，团队要能快速定位是知识层、工具层还是数据层出了错。

## Step 6: 设计 evaluation

至少要定义：

- 3 个质量指标
- 3 个失败信号
- 1 套上线前评测方法
- 1 套运行中抽样评测方法

例如：

- 回答是否引用正确知识
- 工具是否被错误调用
- 动作请求是否在错误条件下被执行

## Step 7: 设计治理清单

这一部分要明确：

- 哪些能力高风险
- 哪些操作需要审批
- 哪些数据敏感
- 哪些调用必须审计
- 哪些结果必须让用户确认后再执行

## Step 8: 用自然语言收束成系统说明

最后，用一段面向团队的自然语言说明，把这个系统讲清楚：

- 用户请求如何进入系统
- 知识如何进入回答过程
- 数据和业务状态如何被访问
- 工具如何被触发
- 观测与评测如何发生
- 高风险动作如何被限制

## Codex 在这个项目里怎么配合

你可以让 Codex：

- 检查你是否混淆了 knowledge / memory / database
- 站在架构评审角度指出缺失的工程层
- 站在治理评审角度指出高风险动作
- 站在评测设计角度指出指标空洞问题

但不要让 Codex 直接替你输出最终架构稿。你真正要练的是工程判断。
