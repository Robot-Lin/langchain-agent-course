# 05 Challenges

这一章的 challenge 目标，不是继续堆概念，而是逼自己做边界判断和系统判断。每道题都建议先自己写答案，再让 Codex 做评审。

## Challenge 1: 边界判断

请分别判断下面需求更偏向：

- memory
- RAG
- 数据库
- MCP / tool / API 接入

### 题目 A

系统记住用户喜欢更短、更行动导向的回答风格。

### 题目 B

系统需要根据最新公司制度文档回答“出差报销”相关问题。

### 题目 C

系统需要读取并更新审批单当前状态。

### 题目 D

系统需要通过一个标准协议访问外部 CRM 能力。

## Challenge 2: 工程分层练习

任选一个你熟悉的 Agent 想法，画出至少 6 层：

- Agent 层
- Tool / MCP 层
- Knowledge 层
- Memory / Data 层
- API / Service 层
- Observability / Evaluation / Governance 层

要求：

- 每一层都写“为什么存在”
- 每一层都写“如果没有这一层，会出什么问题”

## Challenge 3: 指标设计练习

请为一个“企业问答 + 工单动作 Agent”设计：

- 3 个运行指标
- 3 个质量指标
- 3 个失败信号

要求：

- 不能只写泛泛的“准确率高”“用户满意”
- 要尽量让指标能指向具体层次

## Challenge 4: 治理练习

假设你的 Agent 可以：

- 读知识库
- 查客户记录
- 修改工单优先级
- 对外发送消息

请写出一份最小治理清单：

- 哪些动作可以自动执行
- 哪些动作必须确认
- 哪些动作必须审计
- 哪些动作应该完全禁止自动执行

## Challenge 5: 把旧项目工程化

从 Stage 2、Stage 3 或 Stage 4 里选一个你做过的 Agent，回答：

- 它缺哪些工程层
- 如果现在要接真实业务，最大风险在哪里
- 哪一层最先补，为什么

## Challenge 6: MCP 与 API 边界判断

假设你要让 Agent 接企业日历系统、CRM 和内部审批系统，请分别思考：

- 哪些能力更适合暴露成 MCP
- 哪些能力更适合作为内部服务 API
- 哪些能力无论如何都不应该直接暴露给 Agent

## Challenge 7: Evaluation 设计练习

请给一个 RAG Agent 设计一套最小评测方案：

- 上线前怎么测
- 上线后怎么抽样复查
- 什么情况说明系统质量变差了

## Challenge 8: 综合小项目

需求：

设计一个“企业支持 Agent”后端方案。它需要：

- 回答流程问题
- 查询客户状态
- 在人工确认后升级工单
- 输出可追踪、可评估、可治理

交付要求：

- 1 张自然语言架构图
- 1 份层次说明
- 1 份指标方案
- 1 份治理清单

## 自评 Rubric

完成本章 challenge 后，你至少要能做到：

- 不再混淆知识、记忆和业务数据
- 能把工程层按职责拆开
- 能提出具体指标而不是空泛目标
- 能识别高风险动作并设计限制
