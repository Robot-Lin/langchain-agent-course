# 09 Case Comparisons

这一章最适合通过对照看清边界。下面这些案例，帮助你判断某类问题到底属于知识、工具、数据还是治理。

## 案例一: 客服问答 Agent

需求：

- 用户提问产品规则。
- 系统从帮助中心和制度文档中回答。

重点判断：

- 主要能力是 RAG。
- 不是数据库为主，也不是 memory 为主。
- 需要基础 observability 和回答质量评测。

## 案例二: 支持工单 Agent

需求：

- 用户先询问问题原因。
- 系统查询工单、读取客户状态、必要时修改工单流程。

重点判断：

- 同时需要知识层和业务数据层。
- 读取客户状态通常通过 API 或数据库访问。
- 修改工单属于高风险动作，要进入治理层。

## 案例三: 个人学习助理

需求：

- 记住用户学习偏好。
- 根据历史进度调整输出方式。

重点判断：

- memory 是核心。
- 知识库可能是辅助。
- 数据治理相对简单，但仍要区分用户偏好与课程文档。

## 案例四: 企业审批 Agent

需求：

- 汇总信息。
- 读取制度。
- 查询审批状态。
- 在通过确认后触发下一步操作。

重点判断：

- 这是完整工程系统，不是单一 RAG 或单一 tool 系统。
- 需要 API、数据库、治理、审计、observability、evaluation 全套视角。

## 案例五: 为什么很多 Demo 不能上线

常见原因：

- 把文档检索和业务数据混在一起。
- 能调工具，但不知道如何授权和审计。
- 有回答能力，但没有失败追踪。
- 有流程能力，但没有质量评测。

## 本章对照练习

拿你自己的一个想法，分别回答：

- 它的知识从哪里来。
- 它的状态从哪里来。
- 它的数据真相从哪里来。
- 它的动作是怎么被约束的。
- 它的输出是怎么被验证的。

如果这五个问题你都能答清，Stage 5 就真正开始落地了。
