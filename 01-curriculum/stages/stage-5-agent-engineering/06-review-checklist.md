# 06 Review Checklist

这一份文档用于 Stage 5 的阶段放行。不是“看过就算过”，而是确认你是否已经具备把 Agent 接入真实系统的判断力。

## 概念自测

如果下面任意 3 项还答不清楚，建议不要急着进入 Stage 6：

- 我能解释 MCP 在系统中的作用。
- 我能区分 memory、RAG 和数据库。
- 我能说清 API 在 Agent 系统里扮演什么边界角色。
- 我能解释 observability 为什么不能只看最终答案。
- 我能解释 evaluation 为什么必须持续进行。
- 我能指出一个 Agent 的高风险动作在哪里。

## 结构判断自测

你应该已经能做到：

- 给一个 Agent 画出至少 6 层结构。
- 说清每层的输入、输出和职责。
- 指出哪一层承担知识检索，哪一层承担业务读写。
- 指出系统中哪几个地方最容易失控。

## 风险与治理自测

你应该已经能回答：

- 哪些动作可以自动执行。
- 哪些动作必须人工确认。
- 哪些调用必须留下审计记录。
- 哪些敏感数据不应该直接暴露给 Agent。

## 观测与评测自测

你应该已经能设计：

- 至少 3 个运行指标
- 至少 3 个质量指标
- 至少 3 个失败信号
- 一套上线前评测思路
- 一套运行中复核思路

## 表达与复述检查

请尝试不用术语堆砌，而是用自然语言解释：

- 这个 Agent 的知识从哪里来。
- 这个 Agent 的真实业务状态从哪里来。
- 这个 Agent 的动作如何被限制。
- 这个 Agent 出错时团队如何知道问题在哪。

如果你能把这四件事讲清楚，就说明你已经开始具备工程视角。

## 阶段答辩题

### 必答题

1. 为什么 MCP 不只是“另一种工具调用方式”。
2. 为什么 memory、知识库、数据库不能混成一个概念。
3. 为什么 observability 和 evaluation 不是最后再补。
4. 为什么真实 Agent 项目一定要考虑权限与治理。

### 进阶题

1. 为什么很多企业 Agent Demo 一进入真实环境就暴露出边界问题。
2. 如果一个系统只有回答能力，没有 trace、评测和审计，它最容易在什么地方失控。
3. 你会如何把 Stage 4 的复杂任务 Agent 接进真实业务系统。

## 放行标准

通过 Stage 5，至少要满足：

- 完成 1 个工程化架构设计。
- 完成 1 个跟学项目。
- 完成 1 个课后独立项目或综合 challenge。
- 完成 1 次围绕工程边界、评测和治理的正式复盘。

## 进入 Stage 6 前的最后确认

只有当你已经能把 Agent 接上真实系统、知识、数据、观测和治理层时，进入前端体验层才有意义。否则前端只是给脆弱系统套 UI。
