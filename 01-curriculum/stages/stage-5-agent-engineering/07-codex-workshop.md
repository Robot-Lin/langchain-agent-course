# 07 Codex Workshop

这一章里，Codex 最适合扮演的不是“代码生成器”，而是架构评审、边界纠偏器和治理审查员。

## Codex 在 Stage 5 里最适合的 5 个角色

### 1. 边界解释器

适合用来帮你区分：

- MCP 和普通 tool
- memory 和 RAG
- RAG 和数据库查询
- API 边界和底层系统边界

推荐提问：

```text
请你用一个企业支持 Agent 例子，帮我区分 MCP、RAG、memory、数据库和 API 的职责边界。不要直接给结论列表，要一步步解释为什么。
```

### 2. 分层架构评审员

适合在你已经写出初版结构后，让 Codex 找缺层和边界冲突。

推荐提问：

```text
这是我给这个 Agent 画的工程层结构。请你按 Stage 5 的思路，先检查是否缺层，再检查哪些边界混淆了，最后告诉我应该先修哪一层。
```

### 3. 指标与追踪设计教练

适合让 Codex 帮你判断指标是不是空洞。

推荐提问：

```text
请你站在 LangSmith observability 和 evaluation 的视角，检查我这套指标是否真的能反映系统质量。优先指出哪些指标无法定位问题。
```

### 4. 治理与风险审查员

适合让 Codex 站在严格安全评审视角，指出你的方案哪里容易出事故。

推荐提问：

```text
请你站在严格治理评审角度，检查这个 Agent 的工具能力、数据访问和动作执行方案。优先指出最危险的三处设计。
```

### 5. 课后项目导师

课后项目阶段，不要让 Codex 直接生成最终答案，而要让它只做引导与评审。

推荐提问：

```text
我正在做 Stage 5 的课后项目。请不要直接给我完整方案，只根据验收标准检查我的工程分层、知识层、数据层、评测和治理设计，并一步步告诉我下一步该补什么。
```

## 推荐学习顺序

1. 先自己做边界判断。
2. 再让 Codex 帮你纠偏。
3. 先自己画系统层。
4. 再让 Codex 帮你查缺层。
5. 先自己写指标和治理清单。
6. 最后让 Codex 做正式评审。

## 常见错误用法

### 错误一

一上来就让 Codex “帮我设计完整后端架构”。

问题：

- 你会直接跳过最重要的边界判断训练。

### 错误二

只让 Codex 解释概念，不让它评审你的产物。

问题：

- 你会以为自己懂了，但无法迁移到项目设计。

### 错误三

把 challenge 题直接交给 Codex 做。

问题：

- 你失去的恰恰是 Stage 5 最核心的系统判断训练。

## Stage 5 推荐协作闭环

- 你先写边界判断。
- 让 Codex 做概念纠偏。
- 你再画工程层。
- 让 Codex 做架构评审。
- 你再补指标与治理。
- 让 Codex 做最终放行评审。

## 一次完整对练模板

```text
我正在学习 Stage 5: Agent Engineering。
这是我的 Agent 需求和当前结构草案。
请你不要直接替我做完整答案。
先按以下顺序帮助我：
1. 检查 knowledge / memory / database / API / MCP 的边界是否混淆。
2. 指出我缺失的工程层。
3. 指出最重要的三个 observability / evaluation 缺口。
4. 指出最危险的治理问题。
5. 最后只告诉我下一轮该优先修改什么。
```
